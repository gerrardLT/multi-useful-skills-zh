## 问题陈述

GStack 会跨会话运行 30 多个技能，但这些技能之间不会累积任何学习成果。
一次 `/review` 会捕获 N+1 查询模式，而同一个代码库上的下一次 `/review` 又得从零开始。
一次 `/ship` 发现了正确的测试命令，但之后每次 `/ship` 都还得重新发现一遍。
一次 `/investigate` 找到了一个棘手的竞态条件，但未来的任何会话都不会知道这件事。

所有 AI 编程工具都有这个问题。Cursor 有 per-user memory。Claude Code 有 `CLAUDE.md`。Windsurf 有 persistent context。
但它们都不具备复利效应。都不会把学到的东西结构化。也不会让这些知识跨技能共享。

## 我们要构建什么

一种按项目维度沉淀的机构知识，并且它会在会话和技能之间持续产生复利。
这些学习成果是结构化的、带类型的、带置信度评分的，所有 gstack 技能都能读写它们。
目标是：在同一代码库上运行过 20 次会话后，gstack 已经知道所有架构决策、所有历史 bug 模式，以及所有它曾经判断错误的地方。

## 北极星

`/autoship`（Release 5）。一句命令就等于一个完整的工程团队。你描述一个功能，批准计划，后面的事情全部自动推进。
但 `/autoship` 没法在没有 learnings（R1）、review quality（R2）、session persistence（R3）和 adaptive ceremony（R4）的情况下工作。
Release 1-4 就是为了让 `/autoship` 真正成立而打下的基础设施。

## 目标用户

用 AI 做开发的 YC 创始人。那些每周会在真实代码库上运行 gstack 20 次以上，并且能注意到它什么时候重复问了同样问题的人。

## 差异化

| 工具 | 记忆模型 | 范围 | 结构 |
|------|-------------|-------|-----------|
| Cursor | Per-user chat memory | Per-session | Unstructured |
| CLAUDE.md | Static file | Per-project | Manual |
| Windsurf | Persistent context | Per-session | Unstructured |
| **GStack** | **Per-project JSONL** | **Cross-session, cross-skill** | **Typed, scored, decaying** |

## 状态系统

gstack 有 4 个彼此独立的持久化层。它们共享同样的存储模式，也就是都落在 `~/.gstack/projects/$SLUG/` 下的 JSONL / Markdown 文件里，但用途不同：

| 系统 | 文件 | 存储内容 | 写入方 | 读取方 |
|--------|------|---------------|------------|---------|
| **Learnings** | `learnings.jsonl` | 机构知识（坑点、模式、偏好） | 所有技能 | 所有技能（preamble） |
| **Timeline** | `timeline.jsonl` | 事件历史（技能开始/完成、分支、结果） | Preamble（自动） | `/retro`、preamble 上下文恢复 |
| **Checkpoints** | `checkpoints/*.md` | 工作状态快照（决策、剩余工作、相关文件） | `/checkpoint`、`/ship`、`/investigate` | Preamble 上下文恢复、`/checkpoint` 恢复 |
| **Health** | `health-history.jsonl` | 随时间变化的代码质量分数（按工具和综合分） | `/health` | `/retro`、`/ship`（门控）、`/health`（趋势） |

它们并非重叠关系。
Learnings = 你知道什么。
Timeline = 发生过什么。
Checkpoints = 你当前在哪里。
Health = 代码质量怎么样。
每一层回答的是完全不同的问题。

## 发布路线图

### Release 1: `GStack Learns`（v0.13-0.14）- 已发布

**标题：** 每次会话都会让下一次会话更聪明。

已发布功能：
- Learnings 持久化，存放于 `~/.gstack/projects/{slug}/learnings.jsonl`
- `/learn` 技能，用于手动审查、搜索、修剪、导出
- 所有审查发现都增加了置信度校准（1-10 分及显示规则）
- 对观察到的 / 推断出的 learnings 引入置信度衰减（每 30 天衰减 1 分）
- 跨项目 learnings 发现（可选，需经 AskUserQuestion 同意）
- 当审查命中历史 learning 时显示 `Learning applied` 提示
- 集成到 `/review`、`/ship`、`/plan-*`、`/office-hours`、`/investigate`、`/retro`

Schema：
```json
{
  "ts": "2026-03-28T12:00:00Z",
  "skill": "review",
  "type": "pitfall",
  "key": "n-plus-one-activerecord",
  "insight": "Always check includes() for has_many in list endpoints",
  "confidence": 8,
  "source": "observed",
  "branch": "feature-x",
  "commit": "abc1234",
  "files": ["app/models/user.rb"]
}
```

类型：`pattern` | `pitfall` | `preference` | `architecture` | `tool`
来源：`observed` | `user-stated` | `inferred` | `cross-model`

架构采用 append-only JSONL。
重复项在读取时解决，也就是对同一 `key+type` 采用 “latest winner”。
不在写入时进行修改，因此不会引入竞态条件。

### Release 2: `Review Army`（v0.14.3-0.14.4）- 已发布

**标题：** 每个 PR 都有 10 个专家审查员。

已发布功能：
- 7 个并行 specialist subagents：常驻型（testing、maintainability）+
  条件型（security、performance、data-migration、API contract、design）+
  red team（大 diff / 关键发现）
- 结构化 JSON findings，附带置信度分数，并通过指纹在 agent 之间去重
- PR 质量分数（0-10）记录到每次审查，并支持 `/retro` 做趋势分析
- Learning-informed specialist 提示，会把该领域中的历史 pitfalls 注入进去
- 多 specialist 共识高亮，被确认的 findings 会被提升权重
- 通过 `PLAN_COMPLETION_AUDIT` 增强交付完整性
- 重构 checklist：CRITICAL 类别保留在主 pass，其他 specialist 类别拆到 `review/specialists/` 下的聚焦 checklist

### Release 2.5: `Review Army Expansions` - 尚未发布

**标题：** 在 R2 被证明稳定后再发布，重点看核心循环当前表现如何。

前置条件检查：先审查 R2 的质量指标，如 PR 质量分数、specialist 命中率、误报率、E2E 测试稳定性。如果核心循环还有问题，先修核心。

计划发布内容：
- E1：自适应 specialist 门控。对长期 0-finding 的 specialists 自动跳过。通过 gstack-learnings-log 保存 per-project 命中率。用户也可用 `--security` 之类参数强制启用。
- E3：测试桩生成。每个 specialist 输出 findings 的同时生成 `TEST_STUB`。框架由项目自动识别，如 Jest / Vitest / RSpec / pytest / Go test。并接入 Fix-First：`AUTO-FIX` 在修复时顺带创建测试文件。
- E5：跨审查 finding 去重。读取 `gstack-review-read` 中的历史审查条目，抑制那些与用户之前已跳过 findings 完全匹配的结果。
- E7：Specialist 性能跟踪。通过 `gstack-review-log` 记录每个 specialist 的指标。并接入 timeline.jsonl，用于 `/retro` 做趋势分析。

### Release 3: `Session Intelligence`（v0.15.0）- 已发布

**标题：** 你的 AI 会话终于记得之前发生过什么。

已发布功能：
- 会话时间线：每个技能都会自动将开始 / 完成事件写入 `~/.gstack/projects/$SLUG/timeline.jsonl`。仅本地存储，不发送到任何地方，无论遥测设置如何都始终开启。
- 上下文恢复：在压缩后或新会话开始时，preamble 会列出近期 CEO 计划、检查点和审查。Agent 会读取最新一份来恢复上下文。
- 跨会话注入：preamble 会打印当前分支的 `LAST_SESSION` 与 `LATEST_CHECKPOINT`。你甚至在开始输入前，就已经知道上次停在哪里。
- 预测性技能建议：如果最近 3 次会话表现出某种模式（如 review、ship、review），gstack 会建议你下一步可能要做什么。
- 会话开始时显示一条合成的 `Welcome back` 上下文消息。
- `/checkpoint` 技能：保存 / 恢复 / 列出工作快照。支持跨分支列表，便于 Conductor workspace 做 agent 交接。
- `/health` 技能：代码质量记分器，会封装项目内已有工具，如 tsc、biome、knip、shellcheck、tests。输出 0-10 的综合分、趋势追踪，以及分数下降时的改进建议。
- 时间线二进制文件：`bin/gstack-timeline-log` 和 `bin/gstack-timeline-read`
- 路由规则更新：将 `/checkpoint` 和 `/health` 加入 preamble 技能路由

设计文档：`docs/designs/SESSION_INTELLIGENCE.md`

### Release 4: `Adaptive Ceremony` - 尚未发布

**标题：** GStack 尊重你的时间，但不会牺牲安全性。

仪式和信任是两件事。
仪式 = 一个 PR 要经过哪些审查 / 测试 / QA 步骤。
信任 = 一个策略引擎，用来决定适用哪个仪式级别。
两者有关联，但不能混为一谈。

计划发布内容：

**仪式级别：**
- FULL：所有 specialists、对抗性审查、Codex 结构化审查、覆盖率审计、计划完成度。适用于大 diff、新功能、迁移、认证变更。
- STANDARD：对抗性审查 + Codex、覆盖率审计、计划完成度。适用于中等 diff 和常规功能开发。
- FAST：仅对抗性审查。适用于小改动、测试充分、且处于受信任项目中的情况。

**信任策略引擎：**
- 范围感知信任。信任不是全局获得的，而是按变更类别获得。仅文档 PR 上的干净历史，不应换来迁移 PR 上的信任。
- 变更类别检测：docs、tests、config、frontend、backend、migrations、auth、infra。每一类都有自己的信任阈值。
- 信任信号：按类别统计的连续干净审查、`/health` 分数稳定性、回归频率、测试覆盖率趋势。
- 永远不会被快速通道处理的变更类别：migrations、auth/permission changes、new API endpoints、infrastructure changes。无论信任级别多高，它们都必须走 FULL 仪式。
- 信任退化采用渐进式，而不是一次全清零。单个回归不会清空全部信任，只会让对应变更类别降一个等级。

**范围评估：**
- 在 `/review`、`/ship`、`/autoplan` 中，根据 diff 大小、改动文件数和变更类别，将任务分成 TINY / SMALL / MEDIUM / LARGE
- 仪式级别 = f(范围, 信任, 变更类别)

**TODO 生命周期：**
- `/triage`：交互式批准新进 TODOs
- `/resolve`：通过并行 agents 批量处理 TODOs

### Release 5: `/autoship - One Command, Full Feature` - 尚未发布

**标题：** 只描述一个功能，批准计划，剩下全部自动推进。

`/autoship` 不是线性流水线，而是一个可恢复的状态机。
审查和 QA 都可能把工作打回构建 / 修复阶段。
压缩可以在任何阶段发生。
系统必须能优雅恢复。

```text
START
  |
/office-hours
  |
autoplan -> single approval gate
  |
BUILD -> /checkpoint auto-save
  |
/health (score >= 7.0) -> quality gate
  | fail -> back to BUILD
/review
  | ASK items -> back to BUILD
/qa
  | bugs found -> back to BUILD
/ship
  |
/checkpoint archive -> preserve, don't destroy
```

计划发布内容：
- `/autoship` 自主流水线，按上面的状态机运行
  每个阶段都会写入 timeline.jsonl
  每个阶段开始前自动保存检查点
  压缩恢复时，上下文恢复会读取检查点 + 时间线，并从上一个完成的阶段继续
- 完成后对检查点做归档，而不是删除。这样失败的 autoship 运行依然保留恢复状态，便于调试
- `/ideate` 头脑风暴技能（并行发散 agent + 对抗性过滤）
- `/plan-eng-review` 中的研究 agents（代码库分析师、历史分析师、最佳实践研究员、learnings 研究员）

依赖：R1（研究 agents 需要 learnings）、R2（review army 保证质量）、R3（session intelligence 提供持久化）、R4（adaptive ceremony 提供速度）。

### Release 6: `Execution Studio` - 尚未发布

**标题：** 并行执行基础设施。

计划发布内容：

- `/onboard`：通过代码库分析自动生成贡献者指南。
- `/triage-prs`：给维护者做批量 PR 分流。

## 风险登记表

### 把代理信号当成跳过审查的许可
（由 Codex review 于 2026-04-01 提出）

`/health` 分数、干净的审查历史和时间线模式都是有用的信号。
但它们不是安全性的证明。
如果这些信号既用于减少仪式，又被 `/autoship` 依赖，失败模式就会是那种罕见、隐蔽且高严重度的错误。
缓解方式：
- 某些变更类别永远不能快速通道处理（migrations、auth、infra、new endpoints）
- 信任退化采用渐进式，而不是一次性清零
- `/autoship` 在每个项目的第一次运行里一律强制 FULL 仪式。信任必须被慢慢赚到

### 过期的上下文恢复
（由 Codex review 于 2026-04-01 提出）

上下文恢复可能会把错误分支的状态、过时的计划，或者已经无效的检查点注入进来。
缓解方式：
- 检查点的 YAML frontmatter 里写入分支名称。上下文恢复会按当前分支过滤
- 时间线 grep 在展示 `LAST_SESSION` 之前，也会先按分支过滤
- 过期工件检测：如果检查点超过 7 天，就标记为潜在过期，而不直接当成当前状态展示

### 需要验证指标
（由 Codex review 于 2026-04-01 提出）

在发布 R4（Adaptive Ceremony）之前，需要先测量：
- 预测建议准确性（用户是否真的运行了建议技能？）
- 信任策略误跳过率（被快速通道处理的 PR 是否产生了合并后问题？）
- 上下文恢复准确性（恢复出来的上下文是否和真实状态一致？）
- `/health` 分数与实际代码质量的相关性（高分是否真的意味着更少生产事故？）

这些指标应在 R3 使用期间被收集，并在 R4 发布前完成审查。

## 已明确致谢的灵感来源

这条自学习路线图受到了 [Compound Engineering](https://github.com/nicobailon/compound-engineering) 项目中一些想法的启发，作者是 Nico Bailon。
