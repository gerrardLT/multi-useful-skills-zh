# Chrome 侧边栏 + Conductor：我们需要什么

## 我们正在建设什么

现在，当 Claude 在 Conductor 工作区中工作时（编辑文件、运行测试、浏览应用程序），您只能从 Conductor 的聊天窗口中观看。如果 Claude 在您的网站上进行质量检查，您会看到工具调用滚动，但您实际上无法“看到”浏览器。

我们构建了一个 Chrome 侧边栏来解决这个问题。当您运行 `$B connect` 时，Chrome 将打开一个侧面板，其中实时显示 Claude 正在执行的所有操作。你可以在侧边栏中输入消息，克劳德会对它们进行操作——“单击注册按钮”、“转到设置页面”、“总结你所看到的内容”。

问题：侧边栏当前运行其自己单独的 Claude 实例。它看不到主Conductor会话在做什么，主会话也看不到侧边栏在做什么。他们是两个独立的特工，彼此不交谈。

解决方法很简单：使侧边栏成为 Conductor 会话的“窗口”，而不是一个单独的东西。

## 我们需要从 Conductor 那里得到什么（3 件事）

### 1.让我们看看agent在做什么

我们需要一种方法来订阅活动会话的事件。像 SSE 流或 WebSocket 这样的东西会在事件发生时向我们发送事件：

- “克劳德正在编辑 `src/App.tsx`”
- “克劳德正在运行 `npm test`”
- “克劳德说：我会解决 CSS 问题......”

侧边栏已经知道如何呈现这些事件——工具调用显示为紧凑的徽章，文本显示为聊天气泡。我们只需要一条从 Conductor 会话到我们的分机的管道。

### 2. 让我们向会话发送消息

当用户在 Chrome 侧边栏中键入“单击其他按钮”时，该消息应显示在 Conductor 会话中，就像用户在工作区聊天中键入该消息一样。代理在下一个回合中拾取它并对其采取行动。

这是神奇的时刻：用户正在观看 Chrome，发现有问题，在侧边栏中输入更正，然后 Claude 做出响应 - 用户无需切换窗口。

### 3. 让我们从目录创建一个工作区

当 `$B connect` 启动时，它会创建一个用于文件隔离的 git 工作树。我们希望将该工作树注册为 Conductor 工作区，以便用户可以在 Conductor 的文件树中看到侧边栏代理的文件更改。这也为多个浏览器会话奠定了基础，每个浏览器会话都有自己的工作区。

## 为什么这很重要

今天，`/qa` 和 `/design-review` 感觉就像一个黑匣子。 Claude 说“我发现了 3 个问题”，但你看不到它在看什么。将侧边栏连接到 Conductor 后：

- **您观看 Claude 实时测试您的应用程序** — 每次点击、每次导航、每个屏幕截图都会在您观看时显示在 Chrome 中
- **您可以中断** - “不，测试移动视图”或“跳过该页面” - 无需切换窗口
- **一个代理，两个视图** — 编辑代码的克劳德也控制着浏览器。没有上下文重复，没有陈旧状态

## 已经构建了什么（gstack 端）

我们这边的一切都已完成并发货：

- 当您运行 `$B connect` 时自动加载的 Chrome 扩展程序
- 自动打开的侧面板（用户零设置）
- 流事件渲染器（工具调用、文本、结果）
- 带有消息队列的聊天输入
- 重新连接逻辑与状态横幅
- 具有持久聊天历史记录的会话管理
- 代理生命周期（生成、停止、终止、超时检测）

我们这边唯一的变化是：将数据源从“本地 `claude -p` 子进程”交换到“Conductor 会话流”。扩展代码保持不变。

**预计工作量：** 2-3 天 Conductor 工程，1 天 gstack 集成。
