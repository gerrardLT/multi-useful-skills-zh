# 系统化调试

## 铁律

**在完成根因调查前，绝不动手修复。**

只修表象会把调试变成打地鼠。任何没有解决根因的修复，都会让下一个 bug 更难定位。先找到根因，再动手修复。

---

## 第 1 阶段：根因调查

在形成任何假设之前，先把上下文收集完整。

1. **收集症状：** 阅读报错信息、堆栈追踪和复现步骤。如果用户提供的上下文不够，每次只问一个问题。不要一次抛出五个问题。

2. **阅读代码：** 从症状沿着代码路径反向追踪潜在原因。搜索所有引用，阅读故障点附近的逻辑。

3. **检查最近变更：**
   ```bash
   git log --oneline -20 -- <affected-files>
   ```
之前是正常的吗？后来改了什么？如果是回归问题，根因通常就在 diff 里。

4. **复现问题：** 你能稳定触发这个 bug 吗？如果不能，在继续之前先补充更多证据。

5. **检查 memory** 中是否有同一区域之前的调试记录。同一批文件里反复出现 bug，往往说明架构本身存在异味。

输出：**"Root cause hypothesis: ..."**，也就是一个具体、可验证的判断，说明哪里出了问题，以及为什么。

---

## 第 2 阶段：模式分析

检查这个 bug 是否符合已知模式：

**Race condition**：间歇性出现、依赖时序。重点看共享状态的并发访问。

**Nil/null propagation**：比如 `NoMethodError`、`TypeError`。通常是可选值缺少 guard。

**State corruption**：数据不一致、部分更新。检查事务、回调、hooks。

**Integration failure**：超时、意外响应。重点看外部 API 调用和服务边界。

**Configuration drift**：本地正常，staging/prod 出错。检查环境变量、feature flags、数据库状态。

**Stale cache**：显示旧数据，清缓存后恢复。检查 Redis、CDN、浏览器缓存。

另外还要检查：
- 项目中是否已有相关已知问题
- `git log` 中同一区域过去的修复记录。同一批文件重复出问题是架构异味，不是巧合。

**外部搜索：** 如果这个 bug 不符合任何已知模式，就去网上搜索错误类型。 **先脱敏：** 去掉主机名、IP、文件路径、SQL、客户数据。搜索错误类别，不要直接搜原始报错全文。

---

## 第 3 阶段：假设验证

在编写任何修复之前，先验证你的假设。

1. **确认假设：** 在怀疑的根因位置加临时日志、断言或调试输出。重新复现。证据是否吻合？

2. **如果假设是错的：** 去搜索这个错误（先脱敏），然后回到第 1 阶段，继续收集证据。不要猜。

3. **三振规则：** 如果连续 3 个假设都失败，**立刻停止**。告诉用户：

“测试了 3 个假设，但没有一个匹配。这可能是一个架构问题，而不是一个简单的错误。”

   可选方案：
   - 继续以一个新假设调查（说明新假设是什么）
   - 升级为人工评审（需要熟悉该系统的人）
   - 加日志并继续观察（给区域埋点，下次再捕获）

**危险信号：** 如果你看到以下情况，要放慢节奏：
- “先 quick fix 一下再说”：没有所谓“先凑合一下”。要么正确修复，要么升级处理。
- 还没追踪数据流就提出修复方案：这说明你在猜。
- 每修一次都会在别处冒出新问题：这通常是改错层了，不是那段代码本身的问题。

---

## 第 4 阶段：实现

一旦根因被确认：

1. **修根因，不修表象。** 用最小改动消除真正的问题。

2. **保持最小 diff：** 触及尽可能少的文件，改动尽可能少的行。克制住顺手重构相邻代码的冲动。

3. **编写回归测试**，要求：
   - 在没有修复时必须 **失败**（证明测试有意义）
   - 应用修复后必须 **通过**（证明修复有效）

4. **跑完整测试集。** 不允许引入回归。

5. **如果修复涉及超过 5 个文件：** 在继续之前先向用户说明 blast radius。对于 bug 修复来说，这已经算大范围改动。

---

## 第 5 阶段：验证与报告

**重新验证：** 再次复现原始 bug 场景，并确认问题已修复。这一步不是可选项。

运行测试集。

输出结构化调试报告：

**调试报告**
- **Symptom:** 用户观察到的现象
- **Root cause:** 实际根因
- **Fix:** 改了什么，附文件引用
- **Evidence:** 测试输出、复现结果，证明修复生效
- **Regression test:** 新测试的位置
- **Related:** 同一区域的历史问题、架构备注
- **状态：** 完成|已解决问题|封锁

把这份报告按今天的日期保存到 `memory/`，方便后续会话引用。

---

## 重要规则

- **如果修复尝试失败 3 次以上：停止，并反思架构。** 问题往往在架构，而不是单个假设失败。
- **绝不要应用无法验证的修复。** 如果不能复现并确认，就不要发布。
- **绝不要说 "this should fix it."** 必须验证并证明，去跑测试。
- **如果修复涉及超过 5 个文件：** 继续前先告诉用户。
- **完成状态：**
- DONE：已找到根因，已应用修复，已写回归测试，且所有测试通过
- DONE_WITH_CONCERNS：已修复，但无法完全验证（例如间歇性 bug、需要 staging）
- BLOCKED：调查后根因仍不清楚，已升级处理
