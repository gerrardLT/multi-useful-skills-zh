# gstack-中文 目录结构说明

## 项目定位

`gstack-中文` 是一个围绕 AI 编码代理构建的技能与工具仓库。它不只是 skill 集合，还带有浏览器自动化、设计工具、评审体系、升级工具、主机适配层和大量测试。

## 顶层目录与文件

- `README.md`
  - 总入口。
- `ARCHITECTURE.md`
  - 架构说明。
- `ETHOS.md`
  - 项目方法论。
- `DESIGN.md`
  - 设计原则。
- `BROWSER.md`
  - 浏览器能力说明。
- `AGENTS.md`、`CLAUDE.md`
  - 代理协作规范。
- `CHANGELOG.md`
  - 版本记录。
- `SKILL.md`
  - 顶层 skill 入口。
- `package.json`
  - 依赖与脚本入口。
- `bin/`
  - CLI 命令入口。
- `browse/`
  - 浏览器访问与自动化核心模块。
- `design/`
  - 设计生成与对比工具。
- `extension/`
  - 浏览器扩展。
- `hosts/`
  - 不同宿主/平台适配。
- `scripts/`
  - 解析、路由、构建、分析脚本。
- `review/`
  - 代码审查模板与专家规则。
- `test/`
  - 仓库级测试。

## Skills 简要说明

### 自动化与计划审查

- `autoplan`
  - 自动串联多种审查视角，完成计划级自动评审。
- `plan-ceo-review`
  - 用 CEO/创始人视角审查计划。
- `plan-design-review`
  - 用设计视角审查计划。
- `plan-devex-review`
  - 用开发者体验视角审查计划。
- `plan-eng-review`
  - 用工程经理视角审查计划。
- `plan-tune`
  - 调整提问偏好、问题敏感度和 gstack 行为配置。

### 浏览器、设计与文档产出

- `browse`
  - 用无头浏览器做 QA、站点测试和交互验证。
- `design-consultation`
  - 做设计咨询并产出项目设计方向。
- `design-html`
  - 生成高质量原生 HTML/CSS 设计稿。
- `design-review`
  - 发现并修复视觉与交互设计问题。
- `design-shotgun`
  - 一次生成多套设计方案并收集反馈迭代。
- `document-release`
  - 在发布后同步 README、架构文档和变更说明。
- `make-pdf`
  - 把 Markdown 转成出版级 PDF。
- `open-gstack-browser`
  - 启动带侧边栏能力的可视浏览器。

### 质量、安全与发布

- `benchmark`
  - 做性能基准和回归检测。
- `benchmark-models`
  - 对比不同模型运行 gstack 技能的效果与成本。
- `canary`
  - 做部署后的金丝雀监控。
- `cso`
  - 以首席安全官模式做系统安全审计。
- `devex-review`
  - 实测并审查开发者体验。
- `health`
  - 汇总代码质量与仓库健康度。
- `land-and-deploy`
  - 合并、部署并做上线后验证。
- `qa`
  - 做 QA 测试并直接修复问题。
- `qa-only`
  - 只做 QA 测试并输出报告，不修复。
- `review`
  - 做上线前代码与风险审查。
- `ship`
  - 执行版本、变更、PR、发布整套出货流程。

### 调试、协作与状态管理

- `careful`
  - 给危险命令加安全护栏。
- `codex`
  - 提供 Codex 包装模式，用于审查、挑战和咨询。
- `context-restore`
  - 恢复之前保存的工作上下文。
- `context-save`
  - 保存当前工作上下文。
- `freeze`
  - 冻结编辑范围，只允许改指定目录。
- `guard`
  - 组合 `careful` 与 `freeze` 的高安全模式。
- `investigate`
  - 做系统化问题调查和根因分析。
- `learn`
  - 管理项目学习记录和历史经验。
- `office-hours`
  - 做 YC 风格办公时间式产品/项目诊断。
- `pair-agent`
  - 给远程 agent 配对浏览器与访问权限。
- `retro`
  - 做每周工程复盘。
- `unfreeze`
  - 解除编辑冻结范围。

### 安装与环境配置

- `gstack-upgrade`
  - 升级 gstack 到最新版本。
- `landing-report`
  - 查看当前版本槽位、开放 PR 和 WIP 队列。
- `setup-browser-cookies`
  - 把真实浏览器 cookie 导入测试浏览器。
- `setup-deploy`
  - 配置部署平台供后续自动部署使用。
- `setup-gbrain`
  - 配置 gbrain 相关 CLI、存储和接入能力。

## 目录关系

- 一级 skill 目录负责定义工作流。
- `bin/` 负责把能力暴露成命令。
- `scripts/` 负责解析、拼装、生成和校验。
- `hosts/` 负责平台适配。
- `browse/`、`design/`、`extension/` 是较重的功能模块。
- `review/` 与 `test/` 负责质量控制。
- `supabase/` 与 `gstack-upgrade/` 负责服务端支撑与升级迁移。

## 建议阅读顺序

1. `README.md`
2. `ARCHITECTURE.md`
3. `ETHOS.md`
4. `SKILL.md`
5. 目标 skill 目录下的 `SKILL.md`
6. 相关实现模块，如 `browse/`、`design/`、`scripts/`
7. `review/` 和 `test/`

## 常见工作路径

- 想改 skill：
  - 先看对应目录下的 `SKILL.md` 和 `SKILL.md.tmpl`
- 想改命令路由：
  - 看 `bin/` 与 `scripts/resolvers/`
- 想改浏览器能力：
  - 看 `browse/src/` 和 `extension/`
- 想改设计能力：
  - 看 `design/src/`
- 想改平台适配：
  - 看 `hosts/`
- 想查质量规则：
  - 看 `review/` 和 `test/`
