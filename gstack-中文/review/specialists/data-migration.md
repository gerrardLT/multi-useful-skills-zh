# 数据迁移专家审查清单

范围：当 SCOPE_MIGRATIONS=true 时
输出：JSON 对象，每行一个结果。架构：
{“严重性”：“严重|信息","置信度":N,"路径":"文件","行":N,"类别":"数据迁移","摘要":"...","修复":"...","指纹":"路径:行:数据迁移","专家":"数据迁移"}
可选：行、修复、指纹、证据、测试存根。
如果没有发现：输出 `NO FINDINGS` 而没有其他内容。

---

## 类别

### 可逆性
- 此迁移可以回滚而不丢失数据吗？
- 是否有相应的 down/rollback 迁移？
- 回滚实际上是撤消更改还是只是无操作？
- 回滚会破坏当前应用程序代码吗？

### 数据丢失风险
- 删除仍包含数据的列（首先添加弃用期）
- 更改截断数据的列类型 (varchar(255) → varchar(50))
- 删除表而不验证没有代码引用它们
- 重命名列而不更新所有引用（ORM、原始 SQL、视图）
- NOT NULL 约束添加到具有现有 NULL 值的列（需要首先回填）

### 锁定时间
- 在没有 CONCURRENTLY 的情况下对大型表进行 ALTER TABLE (PostgreSQL)
- 在行数超过 100K 的表上添加索引，无需 CONCURRENTLY
- 多个 ALTER TABLE 语句可以合并为一次锁获取
- 在高峰流量时段获取独占锁的架构更改

### 回填策略
- 没有 DEFAULT 值的新 NOT NULL 列（需要在约束之前回填）
- 具有需要批量填充的计算默认值的新列
- 缺少现有记录的回填脚本或 rake 任务
- 回填一次更新所有行而不是批处理（锁定表）

### 索引创建
- 在生产表上不并发创建索引
- 重复索引（新索引覆盖与现有索引相同的列）
- 新外键列上缺少索引
- 部分索引，其中完整索引更有用（反之亦然）

### 多阶段安全
- 必须使用应用程序代码以特定顺序部署的迁移
- 破坏当前运行代码的架构更改（首先部署代码，然后迁移）
- 假设部署边界的迁移（旧代码+新模式=崩溃）
- 缺少在滚动部署期间处理混合旧 /new 代码的功能标志
