# API 合同专家审查清单

范围：当 SCOPE_API=true 时
输出：JSON 对象，每行一个结果。架构：
{“严重性”：“严重|信息”，“信心”：N，“路径”：“文件”，“行”：N，“类别”：“api合同”，“摘要”：“...”，“修复”：“...”，“指纹”：“路径：行：api合同”，“专家”：“api合同”}
可选：行、修复、指纹、证据、测试存根。
如果没有发现：输出 `NO FINDINGS` 而没有其他内容。

---

## 类别

### 重大变化
- 从响应主体中删除了字段（客户端可能依赖它们）
- 更改了字段类型（字符串→数字、对象→数组）
- 添加到现有端点的新必需参数
- 更改了 HTTP 方法（GET → POST）或状态代码（200 → 201）
- 重命名端点而不将旧路径保留为重定向/alias
- 更改了身份验证要求（公开→经过身份验证）

### 版本控制策略
- 在没有版本升级的情况下进行的重大更改（v1 → v2）
- 多个版本控制策略混合在同一 API 中（URL、标头、查询参数）
- 已弃用的端点，没有日落时间表或迁移指南
- 版本特定的逻辑分散在控制器中而不是集中式

### 错误响应一致性
- 新端点返回与现有端点不同的错误格式
- 错误响应缺少标准字段（错误代码、消息、详细信息）
- 与错误类型不匹配的 HTTP 状态代码（200 表示错误，500 表示验证）
- 泄漏内部实现细节的错误消息（堆栈跟踪、SQL）

### 速率限制和分页
- 当类似端点有速率限制时，新端点缺少速率限制
- 分页更改（偏移→光标）没有向后兼容性
- 更改页面大小或默认限制，无需文档
- 分页响应中缺少总数或下一页指示符

### 文档漂移
- OpenAPI/Swagger 规范未更新以匹配新端点或更改的参数
- 描述更改后旧行为的自述文件或 API 文档
- 示例 requests/responses 不再有效
- 缺少新端点或更改参数的文档

### 向后兼容性
- 使用旧版本的客户端：它们会崩溃吗？
- 无法强制更新的移动应用程序：API 仍然适用于它们吗？
- Webhook 有效负载在未通知订阅者的情况下发生更改
- 使用新功能需要更改 SDK 或客户端库
