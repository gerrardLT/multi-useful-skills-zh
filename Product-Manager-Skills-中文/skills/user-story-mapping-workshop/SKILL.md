## 目的
引导产品经理通过自适应提问来创建用户故事地图，问题涵盖系统、用户、工作流和优先级——然后生成一个包含骨干活动、用户任务和发布切片的二维地图。使用此技能可以从扁平的待办列表转向可视化的用户故事地图，以传达全局、识别缺失功能并实现有意义的发布规划——避免“无上下文的混合物”，即故事与整体系统叙述失去联系。

这不是 backlog 生成器，而是一个可视化沟通框架：按用户 workflow 横向组织工作，再按优先级纵向排序。

## 关键概念

### 什么是用户故事地图？

故事地图（Jeff Patton）以**两个维度**组织用户故事：

**水平轴（从左到右）：** 按叙述/工作流顺序排列的活动——即你向他人解释系统时会使用的顺序。

**垂直轴（从上到下）：** 每个活动内的优先级，最核心的任务位于顶部。

**结构：**
```
骨干（活动横贯顶部）
鈫?
用户任务（按优先级垂直向下排列）
鈫?
细节/验收标准（位于底部）
```

### 关键原则

**骨干：** 核心活动构成系统的结构骨架——它们之间不进行优先级排序；它们是叙述流。

**行走骨架：** 所有活动中优先级最高的任务构成最小可行产品——最小的端到端功能。

**肋骨：** 支持性任务在每个活动下垂直向下排列，通过位置指示优先级。

**从左到右、从上到下的构建策略：** 增量式地构建所有主要功能，而不是在开始另一个功能之前完全完成一个功能。

### 为什么这有效
- **视觉沟通：** 故事地图作为信息辐射器持续展示，保持对全局的关注
- **叙述结构：** 按用户工作流组织，而非技术架构
- **发布规划：** 水平切片揭示了 MVP 和增量发布
- **识别差距：** 揭示扁平待办列表所掩盖的缺失功能

### 反模式（它不是什么）
- **不是甘特图：** 故事地图显示优先级，而非时间估算
- **不是技术架构：** 地图遵循用户工作流，而非系统层（UI 鈫?API 鈫?DB）
- **不是项目计划：** 它是发现和沟通工具，而非时间表

### 何时使用
- 启动新产品或主要功能时
- 重构现有待办列表时（从扁平列表转向可视化地图）
- 让利益相关者就范围和优先级达成一致时
- 规划 MVP 或增量发布时

### 何时不该使用
- 单一功能项目（故事地图过于复杂）
- 当待办列表已被充分理解并排定优先级时
- 用于技术重构工作（没有用户工作流可映射）

使用 [`workshop-facilitation`](../workshop-facilitation/SKILL.md) 作为此技能的默认交互协议。

它定义了：

- 进度标签（例如，Context Qx/8 和 Scoring Qx/5）
- 中断处理以及暂停/恢复行为
- 决策点的编号建议

## 应用

此交互技能会提出**最多 5 个自适应问题**，并在每一步提供 **3-4 个编号选项**。

使用 `template.md` 作为议程和输出清单。

交互模式：当你希望获得一个逐步进行、在决策点提供编号建议、并为常规问题提供快速选择选项的流程时，请与 `skills/workshop-facilitation/SKILL.md` 配对。如果用户要求一次性输出，则跳过多轮引导。

### 第 0 步：收集上下文（提问前）

**Agent 建议：**

在我们创建故事地图之前，让我们先收集上下文：

**产品/功能上下文：**
- 你正在映射哪个系统或功能？
- 产品概念、PRD 草稿或现有待办列表
- 网站文案、定位材料或用户流程
- 现有用户故事（如果从扁平待办列表过渡）

**用户上下文：**
- 目标用户画像或用户细分
- 用户研究、访谈或旅程图
- 待完成工作或问题陈述

**你可以直接粘贴此内容，或简要描述系统。**

### 问题 1：定义范围

**Agent 提问：**
“你正在映射什么？（范围是什么？）”

**提供 4 个编号选项：**

1. **整个产品** — “从发现到完成的完整端到端系统”（适用于新产品或完全重写）
2. **主要功能区域** — “更大产品中的特定工作流（例如，‘入职’、‘结账’、‘报告’）”（适用于功能发布）
3. **用户旅程** — “特定的用户目标或待完成工作（例如，‘雇佣承包商’、‘报税’）”（适用于 JTBD 驱动的映射）
4. **重新设计/重构** — “正在重建或简化的现有产品/功能”（适用于遗留系统现代化）

**或描述你的具体范围。**

**用户响应：** [选择或自定义]

---

### 问题 2：识别用户/画像

**Agent 提问：**
“谁是此地图的主要用户？（列出用户画像或用户细分。）”

**提供 4 个编号选项：**

1. **单一画像** — “一种主要用户类型（例如，‘小企业主’）”（简化映射，适用于 MVP）
2. **多个画像，共享工作流** — “不同用户类型，相同核心活动（例如，‘买家’和‘卖家’都浏览列表）”（适用于市场平台）
3. **多个画像，不同工作流** — “不同用户类型，具有不同工作流（例如，‘管理员’ vs. ‘最终用户’）”（需要单独的地图或泳道）
4. **组织内的角色** — “不同的工作职能（例如，‘产品经理’、‘设计师’、‘工程师’）”（适用于内部工具）

**或描述你的用户。**

**调整：** 使用第 0 步提供的上下文中的用户画像（原型画像、JTBD 等）。

**用户响应：** [选择或自定义]

### 问题 3：生成骨干（活动）

**Agent 说：**
“让我们构建骨干——用户为实现目标而执行的活动叙述流。”

**Agent 根据范围（Q1）和用户（Q2）生成 5-8 个活动**，按工作流顺序从左到右排列。

**示例（如果范围 = “电子商务结账”）：**

```
骨干活动（从左到右）：

1. 浏览商品
2. 加入购物车
3. 查看购物车
4. 输入配送信息
5. 输入支付信息
6. 确认订单
7. 接收确认
```

**Agent 提问：**
“这个骨干是否涵盖了完整的工作流？我们应该添加、删除或重新排序活动吗？”

**用户响应：** [批准、修改或添加自定义活动]

---

### 问题 4：生成用户任务（在每个活动下）

**Agent 说：**
“现在让我们在每个活动下添加用户任务，按优先级组织（顶部 = 必须有，底部 = 最好有）。”

**Agent 为每个活动生成 3-5 个用户任务**，按优先级垂直排列。

**示例（对于活动 2：“加入购物车”）：**

```
加入购物车（活动）
鈹溾攢 将单个商品加入购物车（必须有，行走骨架）
鈹溾攢 调整数量（必须有）
鈹溾攢 一次加入多个商品（应该有）
鈹溾攢 保存商品以便稍后购买（最好有）
鈹斺攢 添加礼品包装（最好有）
```

**Agent 对所有骨干活动重复此过程**，显示完整地图。

**Agent 提问：**
“这是否涵盖了关键任务？优先级是否正确（顶部 = MVP，底部 = 后续发布）？”

**用户响应：** [批准、修改或添加自定义任务]

## 常见陷阱

### 陷阱 1：伪装的扁平待办列表
**症状：** 故事地图仅是垂直列表，无水平叙事

**后果：** 丧失沟通优势；仍是“无上下文的碎片”

**修正：** 强制水平结构——顶部为活动，任务垂直向下展开

---

### 陷阱 2：技术架构作为骨干
**症状：** 骨干 = “前端 鈫?后端 鈫?数据库”

**后果：** 非以用户为中心，无法增量交付价值

**修正：** 骨干应遵循用户工作流，而非系统层级

---

### 陷阱 3：功能完备的瀑布模式
**症状：** 发布 1 = “完整构建活动 1”，发布 2 = “完整构建活动 2”

**后果：** 直到所有活动完成前无端到端价值

**修正：** 行走骨架 = 贯穿所有活动的薄切片，逐步增强

---

### 陷阱 4：过早陷入过多细节
**症状：** 试图预先映射所有边缘情况和验收标准

**后果：** 分析瘫痪，丢失全局视野

**修正：** 从骨干 + 高层级任务开始，后续细化

---

### 陷阱 5：地图隐藏在工具中
**症状：** 故事地图存在于 Jira/Miro 中，从未展示

**后果：** 丧失作为信息辐射器的价值

**修正：** 实体打印/张贴地图；使其每日对团队可见
