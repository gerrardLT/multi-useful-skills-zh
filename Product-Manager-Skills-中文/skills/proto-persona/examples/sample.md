# Proto-Persona 示例

## 示例 1：`Manager Mike`（优秀的 Proto-Persona）

```markdown
### 姓名
- Manager Mike

### 简介与人口统计
- 35-42岁，居住在城市/郊区（芝加哥、西雅图、奥斯汀）
- 中型B2B SaaS公司（100-500名员工）的产品总监
- 已婚，育有两个孩子，通勤30分钟车程，重视工作与生活的平衡
- 活跃于LinkedIn和ProductHunt，每年参加1-2场产品经理会议
- 阅读产品经理博客/通讯（如Lenny's, Stratechery），收听播客
- 周末打休闲篮球

### 引用语
- “我花在状态会议上的时间比实际构建产品的时间还多。”
- “我的CEO向我索要我没有的数据，而从工程团队那里获取需要3天。”
- “每个工具都承诺节省时间，但它们都需要一周的上手时间。”

### 痛点
- 每周花费10小时以上在状态会议和撰写更新报告上
- 产品指标没有单一可信来源——数据分散在各个工具中
- 面临加快交付的压力，但团队人手紧张

### 此人试图完成什么？
- 按时交付路线图里程碑，同时避免团队 burnout
- 实时做出数据驱动的优先级决策
- 向高管沟通进展，无需手动报告的开销

### 目标
- 被视为战略思考者，而不仅仅是功能工厂的管理者
- 在两年内晋升为产品副总裁
- 有更多时间陪伴家人（下午6点前下班）

### 态度与影响因素
- **决策权限：** 可批准年度预算1.5万美元以下的工具；超过此额度需副总裁/首席财务官批准
- **决策影响者：** Slack社区中的同行产品经理、前同事、分析师报告（如Gartner）
- **信念与态度：**
  - 对需要开发人员设置的工具持怀疑态度
  - 重视易用性胜过功能深度
  - 偏好能与现有技术栈（Jira, Slack, Figma）集成的工具
  - 愿意为出色的用户体验和支持支付更多费用
```

**为什么这个例子有效：**
- 人口属性具体，并且和行为有关联，而不只是孤立地写 “35-42”
- 引用语能反映真实痛点，不是空泛套话
- 痛点可量化且具体
- 目标同时包含职业动机与个人动机
- 决策权和影响因素都写得很清楚

## 示例 2：糟糕的 Proto-Persona（过于泛化）

```markdown
### 姓名
- John

### 简介与人口统计
- 30-50岁
- 住在某处
- 从事科技行业

### 引用语
- “我想要更好的工具。”

### 痛点
- 工具很糟糕

### 此人试图完成什么？
- 使用好软件

### 目标
- 提高生产力

### 态度与影响因素
- **决策权限：** 也许有
- **决策影响者：** 人们
- **信念与态度：** 喜欢好东西
```

**为什么这个例子失败：**
- 年龄跨度太大，20年范围几乎没信息量
- 没有行为背景，`从事科技行业` 基本等于没说
- 引用语太泛，`更好的工具` 适用于任何用户画像
- 痛点很模糊，`工具很糟糕` 不可执行
- 没有明确目标，也没有明确影响因素

**如何修正：**
- 收窄人群：`35-45岁，总监级，B2B SaaS领域`
- 加上行为细节：`远程优先，活跃于产品经理社区`
- 用真实原话：`我每周花5小时追查状态更新`
- 具体化痛点：`无法了解工程团队正在构建什么`
- 具体化目标：`在两年内晋升为副总裁`
