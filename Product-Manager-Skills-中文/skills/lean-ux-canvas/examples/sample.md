# 精益UX画布示例

### 优秀案例：移动端结账优化

**背景：** 电商公司发现移动端流量已超过桌面端，但移动端转化率仍低15%。

**方框1（业务问题）：**
"移动端流量现占网站访问量的60%，但移动端结账转化率（45%）比桌面端（60%）低15%。我们的结账流程并非为移动端设计：表单字段过多、地址需手动填写、支付分三步完成。竞争对手（如Amazon、Shopify）已支持一键结账。我们正在流失收入。"

**方框2（业务成果）：**
- 在3个月内将移动端结账转化率从45%提升至60%

**方框3（用户）：**
- 每周下单3次以上的移动优先千禧一代（25-35岁）

**方框4（用户成果与收益）：**
- 无需打字即可在30秒内完成结账（避免在移动端键盘上误触带来的挫败感）

**方框5（解决方案）：**
1. 一键结账（Apple Pay、Google Pay）
2. 根据设备位置自动填充地址
3. 为回访客户保存支付方式

**方框6（假设）：**
- "我们相信，若移动优先千禧一代（25-35岁）能通过集成一键Apple Pay实现更快速、无摩擦的结账，移动端结账转化率将从45%提升至60%。"

**方框7（风险最高的假设）：**
- 用户会在看不到逐项费用明细前，就信任一键结账并确认购买

**方框8（实验）：**
- 绿野仙踪测试：先展示一键结账UI，但后台仍走现有支付流程。观察：用户是否会点击“使用Apple Pay支付”？看到Apple Pay弹窗后是否会放弃？

**为何有效：**
- 问题明确（移动端转化率差距）
- 成果可衡量（45% -> 60%）
- 用户群体具体
- 假设可测试
- 实验足够轻量（采用绿野仙踪测试，而非先完整开发）

### 反面案例：功能优先画布（解决方案驱动）

**方框1（业务问题）：**
"我们需要构建一个推荐引擎。"

**为何失败：** 这是解决方案，而非问题。究竟发生了什么变化？为何推荐引擎重要？

**方框2（业务成果）：**
"增加收入。"

**为何失败：** 过于空泛。如何衡量？哪些行为变化代表成功？

**方框5（解决方案）：**
"推荐引擎。"

**为何失败：** 仅有一个解决方案，且是他人预设的方案。完全未探索其他可能性。

**方框6（假设）：**
"我们相信用户会喜欢推荐。"

**为何失败：** 不可测试；未按假设模板撰写；也未将业务成果与用户收益关联。

**应如何改进：**
- 方框1应先描述**变化**（例如“移除追加销售横幅后，平均订单价值下降了20%”）
- 方框2应定义**可衡量的成果**（例如“将平均订单价值从50美元提升至75美元”）
- 方框5应列出**多个解决方案**（例如人工追加销售横幅、AI推荐、捆绑折扣）
- 再分别为每个解决方案撰写假设

### 优秀案例：企业客户入驻摩擦

**方框1（业务问题）：**
"企业客户在6个月后流失，因为入驻需要3周以上的手动配置（SSO、权限、用户导入）。竞争对手提供自助入驻服务。我们的客户成功团队每个客户需花费40小时进行设置，限制了我们的规模化能力。"

**方框7（风险最高的假设）：**
"企业IT管理员可在无人支持的情况下配置SSO。"

**方框8（实验）：**
"礼宾测试：手动引导5家企业客户使用自助入驻向导原型（Figma模型+Loom演示）。衡量：他们能否在3天内完成设置且无需联系支持？"

**为何有效：**
- 问题清晰（手动入驻阻碍规模化）
- 假设可证伪（管理员能否自助服务）
- 实验足够轻量（先进行礼宾测试，再考虑自动化）
