## 目的

引导产品经理创建 **Jeff Gothelf 的 Lean UX Canvas (v2)**。这是一个单页的引导工具，围绕**待解决的业务问题**，而不是**待交付的解决方案**来组织讨论。用它让跨职能团队围绕核心假设对齐，写出可测试的假设，并通过暴露认知缺口（问题、用户、价值，以及为什么这个方案会有效）来确保每个冲刺都有学习产出。
这不是路线图，也不是功能列表，而是一张**“保险单”**：在投入完整开发之前，先把假设变成实验。它把讨论从**产出**转向**成果**，确保团队是在做对的事，而不只是把事做对。

## 关键概念

### 什么是 Lean UX Canvas？

**Lean UX Canvas (v2)** 是一张结构化的单页模板，帮助团队围绕业务问题，而不是围绕解决方案来组织工作。它让跨职能团队在以下方面对齐：
- 存在什么问题（以及为什么现在重要）
- 哪些可衡量的结果代表成功
- 我们在为谁解决问题
- 我们正在做哪些假设
- 我们最需要先学习什么
- 哪些实验可以验证这些假设

**来源：** 由《Lean UX》（O'Reilly, 2013）作者 Jeff Gothelf 创建。Version 2 用来更清楚地区分业务成果与用户成果。
**关键洞察：** 这张 canvas 像一张**保险单**，会在你开始构建前暴露认知缺口，避免把冲刺浪费在错误的问题上。

---

### Canvas 结构（8 个格子）

**布局：** 3 列 x 3 行

```
1. 业务问题 | 2. 业务成果 | 5. 解决方案
3. 用户     | 4. 用户成果 | 5. 解决方案
6. 假设     | 7. 最先学习 | 8. 最小工作 / 实验
```

**8 个格子，建议按这个顺序填写：**

1. **业务问题** - 世界发生了什么变化，导致这里出现了一个值得解决的问题？
2. **业务成果** - 哪些可衡量的行为变化代表成功？
3. **用户** - 你应该先聚焦哪些用户画像？
4. **用户成果与收益** - 用户为什么会追求这个？他们得到什么好处？
5. **解决方案** - 哪些功能/举措可能解决这个问题，并满足用户需求？
6. **假设** - 把 2-5 格连接起来，写成可测试的假设（If/Then 结构）
7. **什么是最需要先学习的？** - 当前最关键、风险最高的假设是什么？
8. **用什么最小工作来学习？** - 验证/证伪这个假设的最小实验是什么？

---

### 为什么这有效

**问题优先，而不是方案优先**
从“世界发生了什么变化？”开始，而不是从“我们应该做 X”开始，避免方案驱动的思维。
**假设驱动**
在开始构建前把假设明确写出来。每个职能都能暴露自己的风险（技术可行性、用户价值、商业可行性）。
**实验聚焦**
先测假设，再投入资源。小实验优于大赌注。
**跨职能对齐**
共享的一张 canvas 会产生共同语言，让所有人看到相同的认知缺口。

---

### 关键区别（避免混淆）

**格子 2（业务成果） vs. 格子 4（用户成果）：**
- **格子 2：** 可衡量的**行为变化**（留存率、网站停留时间、平均订单价值）
- **格子 4：** **目标、收益、情感、共情**（省钱、升职、陪伴家人）
格子 2 是指标。格子 4 是人。
**解决方案（格子 5）是假设，不是承诺**
这里列出的是候选方案（功能、政策，甚至商业模式的变化）。你不是承诺全做，而是在探索解决方案空间。
**假设（格子 6）必须可测试**
使用这个模板：
> “我们相信，如果 [用户] 通过 [解决方案] 获得 [收益]，就能实现 [业务成果]。”
每条假设只聚焦**一个**解决方案。

---

### 反模式（它不是什么）

- **不是功能列表：** 解决方案是待测试的想法，不是待办列表
- **不是项目计划：** Canvas 用来框定学习，不是交付排期
- **不是战略的替代品：** Canvas 负责执行战略，不负责产生战略
- **不是一次性活动：** 随着学习推进，要反复回顾并更新假设

---

### 何时使用

**适合使用的情况：**
- 启动新的产品举措或功能
- 想重构一个现有项目（怀疑自己在做错的事）
- 让跨职能团队围绕假设和实验对齐
- 规划发现冲刺或 MVP
- 利益相关者已经方案驱动（“我们得做 X”），而你需要先暴露假设

**不适合使用的情况：**
- 问题和方案都已验证完（该进入执行了）
- 战术级 bug 修复或技术债（不需要学习）
- 利益相关者无论如何都已拍板某个方案（先处理对齐问题）

完整填写结构见 `template.md`。
这个交互技能会按顺序带你走完 **8 个自适应问题**（每个 canvas 格子一个问题）。在每一步，agent 都会给出引导、示例和选项，帮助你正确填好对应的格子。

### 问题 1：业务问题（格子 1）

**Agent 提问：**

**你现在要解决的业务问题是什么？**

请描述：
- **现状：** 今天这个业务是如何交付价值的？
- **发生了什么变化：** 市场变化、竞争威胁、用户行为变化、新渠道出现
- **为什么重要：** 为什么当前状态已经不符合预期？
**好例子：**
- “我们的移动结账转化率在移动流量超过桌面端后下降了 15%。当前结账流程不是为移动端设计的，而竞品已支持一键结账。”
- “企业客户在使用 6 个月后流失，因为入职需要 3 周以上的人工配置。竞品已提供自助入职。”
**坏例子（太泛）：**
- “我们要提高收入”（没有说明发生了什么变化）
- “用户想要更多功能”（没有说清业务问题）

**Agent 提供 3 个选项：**

1. **我来描述业务问题** - [粘贴或写下你的描述]
2. **帮我识别业务问题** - [Agent 继续追问：发生了什么变化？哪些指标受影响？竞品在做什么？]
3. **我还不知道业务问题是什么** - [Agent 建议先用 `skills/problem-statement/SKILL.md` 或 `skills/problem-framing-canvas/SKILL.md`]

**用户响应：** [选择或描述]

**Agent 验证：** 这里是否描述了**发生了什么变化**，以及为什么它造成了问题？如果没有，就继续追问。

### 问题 2：业务成果（格子 2）

**Agent 提问：**

**你怎么知道这个业务问题被解决了？你要衡量什么？**

重点放在**可衡量的行为变化**上（领先指标也可以）。问自己：
“如果这个方案有效，人们会表现出哪些不同的行为？”

**业务成果示例：**
- 把移动结账转化率从 45% 提高到 60%
- 把企业入职时间从 3 周缩短到 3 天
- 把平均订单价值从 $50 提高到 $75
- 把客户支持工单降低 30%
- 把免费转付费转化率从 5% 提升到 10%

**重要：** 这里是**格子 2（行为变化）**，不是格子 4（用户收益/共情）。指标放这里，情感放格子 4。

**Agent 提供 3 个选项：**

1. **我已经有明确指标** - [说明你的业务成果]
2. **帮我定义可衡量的成果** - [Agent 根据业务问题提建议]
3. **我只有滞后指标（收入、利润）** - [Agent 帮你找到能预测它们的领先指标]

**用户响应：** [选择或描述]

**Agent 验证：** 它们是否可衡量？是否可观察？是否代表了行为变化（而不是泛泛的“增加收入”）？

### 问题 3：用户（格子 3）

**Agent 提问：**

**你应该先聚焦哪些用户类型（也就是用户画像）和客户？**

考虑：
- 谁**买**？
- 谁**用**？
- 谁**配置**？
- 谁**管理**？

**为什么这很重要：** 团队很容易在这里偷懒，说“所有人”。但 canvas 要的是一个**共享的用户视角**，而且不一定就是“购买者”。

**示例：**
- “专业服务领域（顾问、会计师、律师）的中小企业主（2-10 名员工）”
- “负责为 500+ 员工配置 SSO 的企业 IT 管理员”
- “每周点外卖 3 次以上的移动优先千禧一代（25-35 岁）”

**Agent 提供 3 个选项：**

1. **我已经有用户画像** - [参考 `skills/proto-persona/SKILL.md` 或粘贴用户画像]
2. **帮我识别目标用户** - [Agent 追问：谁最痛？谁最可能采用？谁最容易触达？]
3. **我得先创建用户画像** - [Agent 建议先用 `skills/proto-persona/SKILL.md`]

**用户响应：** [选择或描述]

**Agent 验证：** 它是否具体到你能想象出一个真实的人？还是依旧太泛（“所有用户”）？

### 问题 4：用户成果与收益（格子 4）

**Agent 提问：**

**用户为什么会主动寻求你的产品或服务？他们会获得什么好处？我们能看到什么行为变化来证明他们达成了目标？**

这里要聚焦**目标、收益、情感、共情**，不要放指标（指标属于格子 2）。

**用户成果与收益示例：**
- 每周省下 10 小时手工录入时间（可以多陪家人）
- 因为项目交付更快而获得升职
- 避免在朋友面前结账失败的尴尬
- 无需联系支持，也能有信心完成企业软件配置

**为什么这很重要：** 这是**共情桥**。重点是人的动机，而不只是行为变化。

**Agent 提供 3 个选项：**

1. **我知道用户想达成什么** - [描述用户收益]
2. **帮我识别用户收益** - [Agent 提问：他们雇佣这个产品是为了解决什么工作？避免什么痛苦？追求什么收获？]
3. **我得先做研究** - [Agent 建议先用 `skills/jobs-to-be-done/SKILL.md` 或 `skills/discovery-interview-prep/SKILL.md`]

**用户响应：** [选择或描述]

**Agent 验证：** 它是否解释了用户**为什么在乎**，而不只是描述他们会做什么？

### 问题 5：解决方案（格子 5）

**Agent 提问：**

**我们可以做什么，既解决业务问题，又满足客户的需求？**

列出可能有效的**功能、举措、政策、系统，甚至商业模式转变**。鼓励你把解决方案空间拉大：大的、小的、新奇的、甚至非技术性的方案都可以。

**示例：**
- 一键移动结账（Apple Pay、Google Pay）
- 自助入职向导（不用人工配置）
- AI 驱动的推荐引擎
- 管家式入职（高接触、先人工验证，再谈自动化）
- 改定价模式（按使用量收费替代固定费率）

**重要：** 这些是**假设**，不是承诺。你是在探索选项，不是在承诺全做。

**Agent 提供 3 个选项：**

1. **我已经有方案想法** - [列出你的候选方案]
2. **帮我头脑风暴方案** - [Agent 根据问题、成果和用户生成方案]
3. **我脑子里只有一个方案** - [Agent 反问：“如果它不成立，还有什么办法？”]

**用户响应：** [选择或描述]

**Agent 验证：** 你是否至少有 3 个候选方案？（选项越多，后面的假设越好）

### 问题 6：假设（格子 6）

**Agent 提问：**

现在把 2-5 格连起来，写出可测试的假设。

**模板：**

> **我们相信，** [来自格子 2 的业务成果] **将会实现，如果** [来自格子 3 的用户] **通过** [来自格子 5 的解决方案] **获得** [来自格子 4 的收益]。

**规则：**
- 每条假设只针对**一个**解决方案（来自格子 5）
- 需要组合格子 2、3、4、5 的假设
- 必须可测试（你能为它设计实验）

**示例：**

> **我们相信，** 将移动结账转化率从 45% 提高到 60% **将会实现，如果** 移动优先的千禧一代（25-35 岁）**通过** 一键 Apple Pay 集成 **获得** 更快、无摩擦的结账体验。

**Agent 提供：**

基于你的输入，这里是建议的假设（每个解决方案一条）：
1. [生成的假设 1]
2. [生成的假设 2]
3. [生成的假设 3]

**选项：**
- **接受这些假设** - [Agent 记录下来]
- **编辑一条假设** - [修改措辞]
- **自己写假设** - [使用模板]

**用户响应：** [选择或描述]

**Agent 验证：** 每条假设是否清晰表达了：如果方案成立，你相信会发生什么？

### 问题 8：What's the Least Amount of Work We Need to Do to Learn the Next Most Important Thing? (Box 8)

**Agent 提问：**

围绕 Box 7 的最危险假设，设计一个**尽可能快**的实验来验证或证伪它。
**常见实验类型：**
- **Customer interviews** - 5-10 个访谈，验证 value hypothesis
- **Landing page** - Fake door test，测兴趣
- **Concierge / manual prototype** - 先用人工方式交付，再谈自动化
- **Wizard-of-Oz** - 假装功能存在，背后由人操作
- **Smoke test** - 先宣布功能，看报名或点击

**提醒：** 实验设计本身也是门手艺。Canvas 的作用是逼你先做实验，实验本身往往还要继续迭代。
**Agent 提供选项：**

基于你的最危险假设，这里有几个实验选项：
1. **Customer Interviews** - 访谈 10 个用户，问他们是否会使用 [solution]，并观察他们现在的 workaround
2. **Landing Page Test** - 做一个介绍 [solution] 的 landing page，看 signup rate
3. **Concierge Test** - 手工帮 5 个用户交付 [benefit]，观察他们是否得到 [outcome]
4. **Wizard-of-Oz Prototype** - 用人工在幕后“假装” [solution] 已经存在，看看用户能否完成工作流
5. **Design a custom experiment** - [描述你的实验]

## 常见陷阱

### 1. **Starting with Solutions, Not Problems**
**Failure Mode:** Box 1 写成 “We need to build X”，而不是描述发生了什么变化。
**Consequence:** 你只是在做别人已经决定好的方案，却没有验证问题是否存在。
**Fix:** 问：  
“What changed in the world?” 
“Why is this a problem now（而不是 6 个月前）？”
---

### 2. **Vague Business Outcomes**
**Failure Mode:** Box 2 写成 “Increase revenue” 或 “Make users happy”。
**Consequence:** 无法衡量成功，也无法判断实验是否有效。
**Fix:** 用可衡量的行为变化来写。比如：  
“Increase average order value from $50 to $75” 
或 
“Reduce support tickets by 30%.”
---

### 3. **Too-Broad User Segments**
**Failure Mode:** Box 3 写成 “All users” 或 “Everyone.”
**Consequence:** 无法设计有针对性的实验；会把时间浪费在不会采用的人群上。
**Fix:** 先只选一个 persona。后面可以再扩。
---

### 4. **Confusing Box 2 and Box 4**
**Failure Mode:** 把 emotions 写进 Box 2，把 metrics 写进 Box 4（或反过来）。
**Consequence:** hypotheses 会错位，success criteria 也会不清楚。
**Fix:** **Box 2 = Behavior change（metrics）。** **Box 4 = Goals、benefits、emotions（empathy）。**

---

### 5. **Only One Solution in Box 5**
**Failure Mode:** 只写一个 feature，因为 stakeholders 已经拍板。
**Consequence:** 没有探索替代方案，也无法比较哪种 solution 更优。
**Fix:** 强迫自己至少列出 3 个 solutions。问： 
“What else could solve this problem?”
---

### 6. **Skipping Experiments (Box 8)**
**Failure Mode:** “我们先做出来再看会发生什么。”
**Consequence:** 很容易浪费几周甚至几个月，做出错误的东西。
**Fix:** 先设计最小实验。如果想不到，就用 `skills/pol-probe-advisor/SKILL.md` 来选验证方法。
