## 目的
引导产品经理创建机会解决方案树（OST）：从利益相关者的请求中提取目标成果，生成多个机会选项（待解决的问题），映射潜在解决方案，并基于可行性、影响力和市场契合度选择最佳的概念验证（POC）。用它将模糊的产品需求转化为结构化的探索过程，确保团队在跳向解决方案之前先确认自己解决的是正确的问题，避免陷入“功能工厂”和过早收敛。
这不是路线图生成器，而是一个结构化的探索过程，产出的是经过验证的问题机会和可测试的解决方案假设。
## 关键概念

### 什么是机会解决方案树（OST）？

OST 是一个可视化框架（Teresa Torres，《持续探索习惯》），用于连接：
1. **期望成果**（业务目标或产品指标）
2. **机会**（可能驱动该成果的客户问题、需求、痛点或渴望）
3. **解决方案**（解决每个机会的方式）
4. **实验**（验证解决方案的测试）

**结构：**
```text
         Desired Outcome (1)
                |
    +-----------+-----------+
    |           |           |
Opportunity  Opportunity  Opportunity (3)
    |           |           |
  +-+-+       +-+-+       +-+-+
  | | |       | | |       | | |
 S1 S2 S3    S1 S2 S3    S1 S2 S3 (9 total solutions)
```

### 为什么这有效
- **成果驱动：** 从业务目标开始，而不是从功能需求开始
- **先发散，再收敛：** 先探索多个机会，再选择解决方案
- **问题聚焦：** 机会必须是问题，不能是伪装成问题的解决方案
- **可测试：** 每个解决方案都要映射到实验，而不是“做了就发布”
- **POC 选择：** 在投入资源前先比较可行性、影响力、市场契合度

### 反模式（它不是什么）
- **不是功能列表：** 机会是客户面临的问题，不是“我们需要暗黑模式”
- **不是解决方案优先：** 不要从“我们应该做X”开始，要从“客户在Y上有困难”开始
- **不是瀑布式规划：** OST 是探索工具，不是项目计划
- **不是一次性工作：** OST 会随着实验结果持续演进

### 何时使用
- 利益相关者提出了一个功能需求或产品倡议
- 要为新的产品领域启动探索
- 要澄清模糊的 OKR 或战略目标
- 要决定先解决哪些问题
- 要让团队对目标成果达成一致

### 何时不要使用
- 问题已经验证完毕（应转向解决方案测试）
- 战术性 bug 修复或技术债务（不需要探索）
- 利益相关者坚持指定某个解决方案（先处理对齐问题）

## 应用

完整填写结构化 `template.md`。
这个交互式技能分两阶段执行：
**阶段 1：** 生成 OST（提取成果、识别机会、映射解决方案）
**阶段 2：** 选择 POC（评估解决方案，推荐最佳起点）

## 阶段 1：生成机会解决方案树

### 问题 1：提取期望成果

**Agent 提问：**
“这个倡议的期望成果是什么？（你想推动哪个业务或产品指标？）”

**提供 4 个枚举选项：**

1. **收入增长** - “增加 ARR、扩大现有客户收入、创造新收入来源”（常见于规模化产品）
2. **客户留存** - “减少流失、提高激活率、改善参与度/粘性”（常见于已有留存问题的成熟产品）
3. **客户获取** - “增加注册、试用转化、新用户增长”（常见于早期或增长型产品）
4. **产品效率** - “降低支持成本、缩短价值实现时间、改善运营指标”（常见于成熟产品做效率优化）
**或者直接描述你的具体成果（要可衡量，例如“将试用到付费转化率从 15% 提高到 25%”）。**

**用户响应：** [选择或自定义]

**Agent 提取并确认：**
- **期望成果：** [具体、可衡量的成果]
- **为什么重要：** [来自利益相关者请求或上下文的理由]

---

### 问题 2：识别机会（待解决的问题）

**Agent 生成 3 个机会**，基于期望成果和你提供的背景。
**Agent 说：**
“基于你的期望成果（[来自问题 1]）和提供的背景，这里有 **3 个机会**（客户问题或需求），它们都可能推动这个成果：

**示例（如果成果 = 提高试用到付费转化率）：**

1. **机会 1：用户在试用期间未体验到价值** - “新用户注册后没有完成引导，也未达到‘顿悟时刻’，在看到核心价值前就流失了。”
   - 证据：[来自上下文：引导分析、支持工单、退出调查]

2. **机会 2：定价不清晰或错位** - “用户不确定付费计划是否值得；不理解这个价格对应什么价值；定价页面本身也可能令人困惑。”
   - 证据：[来自上下文：定价页面跳出、销售异议]

3. **机会 3：免费计划‘足够好’** - “用户长期停留在免费层级，因为已经够用；没有足够强的升级理由。”
   - 证据：[来自上下文：免费增值留存、支持工单中的变通方法]

**Agent 提问：**
“你觉得哪个机会最值得先展开，还是你想修改/补充机会？”

**用户响应：** [选择或自定义]

---

### 问题 3：为选定的机会生成解决方案

**Agent 生成 3 个解决方案想法**，针对你选中的机会。
**Agent 说：**
“针对 **机会 [X]: [名称]**，这里有 **3 个可测试的解决方案**：

**示例（如果选中机会 1：用户在试用期间未体验到价值）：**

1. **解决方案 1：引导式入门清单** - “用交互式清单逐步引导用户完成核心工作流（例如‘创建你的第一个项目’、‘邀请队友’、‘完成一个任务’）。”
   - 假设：结构化引导能提高完成率。
   - 实验：A/B 测试清单 vs. 无清单，测量激活率。

2. **解决方案 2：价值实现触发器** - “当用户卡住时主动提示（例如‘你还没创建项目，这里有个模板可以直接开始’）。”
   - 假设：主动提示能减少中途流失。
   - 实验：跟踪提示参与度，测量试用到付费提升。

3. **解决方案 3：人工辅助引导** - “为高价值试用用户（企业版、团队计划）提供 15 分钟的引导通话。”
   - 假设：人工高接触能提高高意向用户的转化率。
   - 实验：为 50 个试用用户提供该服务，与对照组比较转化率。

**Agent 提问：**
“你想继续为另一个机会生成解决方案，还是直接进入 POC 选择？”

**用户响应：** [选择或自定义]

## 阶段 2：选择概念验证（POC）

生成解决方案后，Agent 会帮助你决定先测试哪一个。
---

### 问题 4：评估解决方案（可行性、影响力、市场契合度）

**Agent 说：**
“我们来评估这些解决方案，选出最佳起点。我会按 **可行性**（多难做）、**影响力**（能在多大程度上推动成果）和 **市场契合度**（与客户需求匹配度）给每个解决方案打分。”

**Agent 生成评估表：**

| 解决方案 | 可行性 (1-5) | 影响力 (1-5) | 市场契合度 (1-5) | 总分 | 理由 |
|----------|-------------------|--------------|------------------|-------------|-----------|
| 解决方案 1：引导式入门清单 | 4 | 4 | 5 | 13 | 可行性高（成熟 UI 模式）、影响力已被行业验证、市场契合度强（用户本就期待引导） |
| 解决方案 2：价值实现触发器 | 3 | 3 | 4 | 10 | 可行性中等（依赖分析集成）、影响力中等（取决于触发质量）、市场契合度尚可 |
| 解决方案 3：人工辅助引导 | 5 | 5 | 3 | 13 | 极高可行性（几乎不用开发）、影响力高（人工服务效果强）、但不够可扩展 |

**评分标准：**
- **可行性：** 1 = 需要数月，5 = 几天到几周
- **影响力：** 1 = 对成果影响很小，5 = 很可能显著推动成果
- **市场契合度：** 1 = 客户并不在意，5 = 客户明确有这个需求
**Agent 推荐：**

**推荐 POC：解决方案 1（引导式入门清单）**

**为什么选这个 POC：**
- 在可行性（4-8 周可落地）和影响力（成熟模式）之间取得了平衡
- 市场契合度强（用户通常期待引导）
- 假设可测试：“如果我们引导用户完成核心工作流，激活率将从 X% 提升到 Y%”
**备选 POC：** 解决方案 3（人工辅助引导）
- 如果你想以零开发工作快速学习，可以先做这个
- 手动服务 20-50 个试用用户，收集定性反馈
- 再把学到的东西用于自动化方案（解决方案 1）
**不推荐作为 POC：** 解决方案 2（价值实现触发器）
- 前置投入更高
- 影响力很依赖触发质量（本身又需要多轮实验）
- 更适合作为验证核心引导流程之后的下一步
---

### 问题 5：定义实验

**Agent 提问：**
“你打算怎么测试这个解决方案？实验是什么？”

**提供 3 个枚举选项：**

1. **A/B 测试** - “做一个 MVP，给 50% 的试用用户展示，并与对照组比较转化率”（适合：量化验证，前提是流量足够）
2. **原型 + 可用性测试** - “做一个可点击原型，看 10 个用户尝试完成引导，收集定性反馈”（适合：早期验证，流量少也能做）
3. **人工服务测试** - “先人工运行这个解决方案给 20 个用户（例如亲自带引导），再测结果”（适合：快速学习、无需开发）

**或者直接描述你的实验方法。**

**用户响应：** [选择或自定义]

## 常见陷阱

### 陷阱 1：机会伪装成解决方案
**症状：** “机会：我们需要一个移动应用”
**后果：** 你其实已经收敛到解决方案了，根本没展开问题空间。
**修复：** 把机会改写成客户问题：
“移动优先用户无法在移动中访问产品。”
---

### 陷阱 2：跳过发散（直接跳到一个解决方案）
**症状：** “我们知道解决方案就是 [X]，直接做就行”
**后果：** 会错过更优方案，也没有真正学习。**Fix:** 每个 opportunity 至少生成 3 个 solutions。先发散，再收敛。

---

### 陷阱 3：Outcome is Too Vague
**Symptom:** “Desired Outcome: Improve user experience”
**Consequence:** 无法衡量成功，也无法比较 opportunities。
**Fix:** 让 outcome 可衡量：
“Increase NPS from 30 to 50”
或
“Reduce onboarding drop-off from 60% to 40%.”

---

### 陷阱 4：No Experiments (Just Build It)
**Symptom:** 选完 solution 就直接扔进 roadmap

**Consequence:** 没验证，极高概率做错。
**Fix:** 每个 solution 都必须映射到一个 experiment。没有实验，就不算完整 OST。

---

### 陷阱 5：Analysis Paralysis (Exploring Forever)
**Symptom:** 生成了 20 个 opportunities、50 个 solutions，却一直不选
**Consequence:** 团队卡在 discovery，没有进展。
**Fix:** 限制为 3 个 opportunities，每个 3 个 solutions（共 9 个）。选一个 POC，跑实验，学习，再迭代。
