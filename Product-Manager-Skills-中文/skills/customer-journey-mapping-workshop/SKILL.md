## 目的
通过询问关于行为者（persona）、场景/目标、旅程阶段、动作/情绪以及改进机会的自适应问题，指导产品经理创建客户旅程地图。用此来可视化端到端的客户体验，识别痛点，并在团队间建立共享的心智模型——避免肤浅的功能列表，确保发现工作聚焦于真实的客户问题，而非假设的解决方案。

这不是功能路线图，而是一个用于发现问题和推动对齐的工具。它帮助你找出体验在哪些地方断裂，以及哪些改进最值得优先投入。

## 关键概念

### 什么是客户旅程地图？
旅程地图（NNGroup）可视化了“一个人为完成目标所经历的过程”。它将用户行为编译成时间线，融入想法和情绪以创建叙事，然后浓缩并打磨成一个可视化的产物。

### 五个关键组成部分（NNGroup框架）
1. **行为者** — 锚定地图视角的特定 persona 或用户
2. **场景 + 期望** — 情境背景及相关目标
3. **旅程阶段** — 组织体验的高级阶段（例如，发现、试用、购买、使用、寻求支持）
4. **动作、心态和情绪** — 用户在整个阶段中的行为、想法和情绪反应
5. **机会** — 识别体验可以在何处改进的洞察

### 旅程地图结构
```
行为者: [Persona 名称]
场景: [目标/背景]

阶段 1: 发现 鈫?阶段 2: 试用 鈫?阶段 3: 购买 鈫?阶段 4: 使用 鈫?阶段 5: 支持
   鈫?                 鈫?               鈫?              鈫?              鈫?
动作:           动作:         动作:        动作:        动作:
想法:          想法:        想法:       想法:       想法:
情绪: 馃槉馃槓馃槥    情绪:        情绪:       情绪:       情绪:
   鈫?                 鈫?               鈫?              鈫?              鈫?
机会:     机会:   机会:  机会:  机会:
```

### 为什么这有效
- **强制对话：** 团队就客户体验的共享理解达成一致
- **揭示痛点：** 情绪 + 动作突出了体验在何处断裂
- **优先排序改进：** 按影响力排序的机会指导路线图决策
- **以人为中心：** 聚焦于客户视角，而非内部流程

### 反模式（它不是什么）
- **不是服务蓝图：** 旅程地图聚焦于客户视角；服务蓝图映射内部运营
- **不是用户故事地图：** 旅程地图支持发现；用户故事地图促进实施规划
- **不是体验地图：** 旅程地图针对特定用户和产品；体验地图探索更广泛的人类行为

### 何时使用
- 开始客户发现（理解当前体验）
- 为留存/参与举措识别痛点
- 让跨职能团队在客户视角上达成一致
- 优先排序要首先解决的问题

### 何时不该使用
- 当你已经深入了解客户旅程时
- 用于技术重构（没有面向客户的旅程）
- 作为用户研究的替代品（地图需要研究输入）

### 交互的事实来源

使用 [`workshop-facilitation`](../workshop-facilitation/SKILL.md) 作为此技能的默认交互协议。

它定义了：
- 会话预告 + 进入模式（引导式、上下文倾倒、最佳猜测）
- 使用通俗语言提示的单问题轮次
- 进度标签（例如，上下文 Qx/8 和评分 Qx/5）
- 中断处理和暂停/恢复行为
- 决策点的编号建议
- 常规问题的快速选择编号响应选项（在有用时包含 `其他（请说明）`）

此文件定义了领域特定的评估内容。如有冲突，遵循此文件的领域逻辑。

## 应用

此交互技能提出**最多 5 个自适应问题**，在每个步骤提供 **3-4 个枚举选项**。

交互模式：当你想要一个逐步进行的流程，在决策点提供编号建议，并为常规问题提供快速选择选项时，与 `skills/workshop-facilitation/SKILL.md` 配对使用。如果用户要求一次性输出，则跳过多轮引导。

### 问题 1：识别行为者（Persona）

**Agent 提问：**
“这个旅程地图的行为者是谁？（哪个 persona 或用户细分？）”

**提供 4 个枚举选项：**

1. **主要 persona** — “你的主要目标客户（例如，‘小企业主’）”（最常见的起点）
2. **次要 persona** — “具有不同需求的次要用户细分（例如，‘企业管理员’ vs. ‘最终用户’）”（当主要 persona 已映射时使用）
3. **高流失率 persona** — “流失率最高的用户细分（例如，‘未转化的试用用户’）”（适用于留存举措）
4. **新发现的 persona** — “近期研究中出现的新兴用户细分（例如，COVID 后的‘远程团队’）”（适用于市场扩张）

**或描述你的具体 persona。**

### 问题 3：识别旅程阶段

**Agent 说：**
“让我们将旅程分解为高级阶段（通常从开始到结束有 4-6 个阶段）。”

**Agent 根据场景（Q2）生成 4-6 个旅程阶段。**

**示例（如果场景 = “首次使用”）：**

```
旅程阶段（从左到右）：

1. 发现 — 用户了解产品
2. 评估 — 用户研究、比较替代方案
3. 试用 — 用户注册，开始引导
4. 激活 — 用户达到“顿悟时刻”，体验价值
5. 使用 — 用户将产品集成到工作流中
6. 扩展 — 用户考虑升级或邀请团队
```

**Agent 提问：**
“这些阶段是否捕捉了完整的旅程？我们应该添加、删除或重命名阶段吗？”

**用户响应：** [批准或修改]

### 问题 4：映射每个阶段的动作、想法、情绪

**Agent 说：**
“现在让我们映射行为者在每个阶段所做的、所想的和所感受的。”

**Agent 根据上下文（步骤 0）和场景（Q2）为每个阶段生成 3-5 个动作、想法和情绪。**

**示例（针对阶段 3：“试用 — 用户注册，开始引导”）：**

```
阶段 3: 试用（引导）

动作：
- 使用邮箱注册
- 收到欢迎邮件
- 首次登录
- 看到空仪表板
- 搜索“入门”指南

想法：
- “这看起来很有前途，但我不确定从哪里开始”
- “我需要看教程视频吗？”
- “第一步是什么？”

情绪：
- 好奇但不确定 馃?
- 略感沮丧（没有明确的下一步）馃?
- 希望它会变得更容易 馃?

痛点：
- 没有引导清单或导览
- 空状态没有建议下一步操作
- 导航中选项太多（令人不知所措）
```

**Agent 提问：**
“这是否准确捕捉了客户体验？我们应该调整动作、想法或情绪吗？”

### 问题 5：识别机会（需解决的痛点）

**Agent 说：**
“基于旅程地图，让我们识别改进机会——按影响力排序。”

**Agent 生成 5-7 个机会**（情绪强度最高或流失率最高的痛点）。

**示例：**

```
# 机会（按影响力排序）

## 1. 引导缺乏引导性的第一步（阶段 3：试用）
**痛点：** 用户看到空仪表板，不知道首先做什么
**证据：** 60% 的注册用户在 24 小时内未完成第一个操作
**机会：** 添加交互式引导清单（“创建你的第一个项目”、“邀请队友”）
**影响：** 高 — 直接影响激活率

---

## 2. 定价页面令人困惑（阶段 2：评估）
**痛点：** 用户不理解哪个计划适合他们的需求
**证据：** 定价页面跳出率高（70% 的人未注册就离开）
**机会：** 添加计划比较工具或“哪个计划适合我？”测验
**影响：** 高 — 直接影响试用转化率

---

## 3. 支持难以找到（阶段 5：使用）
**痛点：** 用户遇到问题，难以找到帮助
**证据：** 支持工单常说“我在文档中找不到答案”
**机会：** 添加应用内帮助小部件、上下文工具提示
**影响：** 中 — 影响留存，但较少用户进入此阶段

---

## 4. 邮件确认缺乏上下文（阶段 1：发现）
**痛点：** 营销邮件没有清晰解释价值
**证据：** 邮件活动点击率低（5% vs. 行业平均 15%）
**机会：** 用客户语言重写邮件，明确行动号召
**影响：** 中 — 影响漏斗顶部认知

---

## 5. 升级提示感觉咄咄逼人（阶段 6：扩展）
**痛点：** 用户认为升级提示是推销，而非帮助
**证据：** NPS 评论中的负面情绪（“太多升级弹窗”）
**机会：** 在上下文中显示升级价值（当用户达到免费计划限制时）
**影响：** 低 — 影响较小的用户子集
```

**Agent 提问：**
“这些机会是否符合你的优先级？我们应该首先关注哪些？”

**用户响应：** [选择或自定义]

## 示例

### 示例 1：良好的旅程地图（SaaS 产品入门）

**Q1 回答：** "主要用户画像 — 小企业主"

**Q2 回答：** "首次使用 — 新用户入门，从发现到激活"

**Q3 - 生成的阶段：**
```
发现 → 评估 → 试用 → 激活 → 使用 → 扩展
```

**Q4 - 阶段 3（试用）映射：**

```
行为：
- 通过 Google SSO 注册
- 收到欢迎邮件
- 登录，看到空白仪表板
- 点击“帮助”按钮，观看 5 分钟教程
- 尝试创建第一个项目，在表单处卡住

想法：
- “这看起来挺简单的”
- “等等，‘工作区’和‘项目’有什么区别？”
- “我需要填写所有这些字段吗？”

情绪：
- 最初很兴奋 馃槉
- 对术语感到困惑 馃槙
- 因表单不清晰而沮丧 馃槥

痛点：
- 没有引导式入门清单
- 术语未解释（工作区 vs. 项目）
- 表单一开始就要求太多必填字段
```

**Q5 - 识别的机会：**
1.  添加入门清单（高 — 影响激活）
2.  简化术语（中 — 影响理解）
3.  减少必填表单字段（中 — 影响完成率）

**为何有效：**
- 情绪 + 行为清晰地揭示了痛点
- 机会与特定阶段挂钩
- 有研究证据支持（流失数据、支持工单）

### 示例 2：糟糕的旅程地图（过于笼统）

**阶段：“使用产品”**

**行为：**
- 使用产品
- 完成任务

**想法：**
- “这很好”

**情绪：**
- 开心 馃槉

**为何失败：**
- 缺乏具体性（什么任务？哪些功能？）
- 未识别痛点（一切都“很好”）
- 无法提取可操作的机会

**修复方法：**
- 具体化：“用户创建发票 → 发送给客户 → 跟踪付款状态”
- 包含真实客户引述：“我希望我能批量发送发票”
- 展示情绪的高峰和低谷（不仅仅是开心）

## 常见陷阱

### 陷阱 1：映射内部流程，而非客户体验
**症状：** 旅程阶段 = “线索生成 → 资格审核 → 安排演示 → 成交”

**后果：** 关注销售流程，而非客户视角

**修复方法：** 从客户视角映射：“发现问题 → 研究解决方案 → 试用产品 → 采纳”

---

### 陷阱 2：没有情绪或痛点
**症状：** 旅程地图仅列出行为，没有想法/情绪

**后果：** 错失重点——无法识别体验在何处中断

**修复方法：** 添加客户引述、情绪状态（沮丧、高兴、困惑）

---

### 陷阱 3：一张地图中包含太多用户画像
**症状：** 试图在一张旅程地图中映射“所有用户”

**后果：** 失去焦点，变得笼统

**修复方法：** 每个用户画像一张地图。如果有多个用户画像，请创建单独的地图。

---

### 陷阱 4：机会未排定优先级
**症状：** 列出 20 个机会但没有排序

**后果：** 团队陷入瘫痪，不知从何开始

**修复方法：** 根据证据和情绪强度，按影响（高/中/低）排序。

---

### 陷阱 5：地图在孤立中创建
**症状：** 产品经理独自创建旅程地图，未让团队参与

**后果：** 没有共享心智模型，地图无法驱动决策

**修复方法：** 与跨职能团队（产品、设计、工程、支持）共同举办工作坊。
