# 仓库指南

## 工作哲学：教学性与实用性并重

这个仓库不仅为代理添加技能，它还承担着另一层使命：**帮助产品经理自身变得更强大，并将经验持续传递下去。**

这里的技能同时服务于两个对象：

- 使用技能的产品经理
- 执行技能的 AI 代理

产品经理在使用后，应该比开始时懂得更多，而不仅仅是获得一个完成产物。技能中包含的推理过程，决定了输出是否值得信赖、是否可解释、是否可传递。

**ABC - Always Be Coaching** 是核心治理原则之一。

这直接影响你在此仓库中作为代理的工作方式：

- **不要为了简洁而牺牲解释。**
- **反模式是承重结构。** 请勿删除。
- **示例要展示推理过程，而不仅仅是展示结果。**
- **始终同时服务双重受众：** 一边是建立判断力的人类产品经理，一边是执行任务的 AI 代理。

如果你在进行“效率优化”，请删除的是冗余内容，而不是经验教训。

## 项目结构与模块组织

- `skills/<skill-name>/SKILL.md`：每个技能的主文件，目录名使用小写短横线命名法
- `commands/<command-name>.md`：串联本地技能的可复用命令
- `catalog/`：用于快速浏览的生成索引
- `research/`：支撑技能的参考文章
- `docs/`：使用指南
- `app/`：Streamlit（测试版）交互环境
- `dist/`：生成的发布产物，请勿直接修改
- 根目录文档如 `README.md`、`CONTRIBUTING.md`、`PLANS.md`、`CLAUDE.md`：说明贡献流程、目录结构、提炼方法

## 构建、测试与开发命令

这是一个以 Markdown 为主的仓库，仅包含轻量级的校验与打包脚本。

- `rg --files`：快速列出文件
- `rg "SKILL.md"`：查找技能定义
- `rg "skill-name"`：提交前验证引用
- `./scripts/find-a-command.sh --list-all`：列出所有命令
- `./scripts/validate-skills.sh`：校验标准技能目录
- `./scripts/build-release.sh`：构建发布产物
- `./scripts/test-library.sh`：验证技能与命令并重建目录
- `streamlit run app/main.py`：启动 Streamlit（测试版）交互环境

## 编写风格与命名规范

- 使用 Markdown，标题清晰，段落简短
- 技能必须包含标准章节：
  - 目的
  - 核心概念
  - 应用
  - 示例
  - 常见陷阱
  - 参考资料
- 每个技能顶部必须有 frontmatter：`name`、`description`、`type`
- `name` 不超过 64 个字符
- `description` 不超过 200 个字符
- 技能文件夹名必须与 frontmatter 中的 `name` 完全一致
- 命令与模板示例请使用带语言标签的代码块
- 表达要简洁、有立场，避免使用填充词

### 交互式技能

**定义：**  
通过连续提问来收集上下文，并据此给出智能下一步建议的多轮对话型流程。

**特征：**
- 一次问一个问题，或少量分批提问
- 后续问题受前面回答影响
- 给出**带编号、结合上下文的推荐**
- 用户可以选择数字，也可以自定义输入
- 会随着用户选择而调整

## 测试指南

没有自动化测试框架，因此请通过以下方式验证：

- 确保技能路径引用可解析
- 核对示例和参考资料的准确性
- 检查结构是否合规、是否易读
- 对于 Claude Web 上传场景，确认 frontmatter 是合法的 YAML

## 工作原则：先吃自己的狗粮

- 做结构性决策前，优先使用仓库自身的定义、脚本和标准
- 判断技能类型/分类时，先参考本地 `README.md`、`CLAUDE.md` 和相关 `SKILL.md`
- 优先使用仓库工具进行验证，而不是凭感觉
- `skills/` 是所有打包工作的权威数据源
- 兼容 Claude Desktop/Web、Claude Code 和 Codex
- 脚本优先使用简单、可维护的 Bash
- 非必要不重命名技能

## 跨仓库边界

- 这个仓库是共享的产品经理技能库，不是 Productside 剧本分发仓库
- Productside 的剧本技能内容应在 `/Users/deanpeters/Code/productside_playbook_skills` 中维护
- 支持 Productside 工作时，默认将本仓库视为只读参考，除非明确要求修改

## Claude 自定义技能兼容性

- Claude Web 上传要求文件名是 `Skill.md`（大小写敏感）
- 技能内脚本必须是确定性的、避免网络调用，并在技能中写明用途
- 分享或运行前请先审查安全性
- 不要删除已有的 Claude Code marketplace 文件，例如 `.claude-plugin/marketplace.json`

## Codex 技能指南

当用户寻求产品管理帮助时，先查看可用技能，再做决定，不要直接即兴发挥。

常见相关主题包括：

- 探索
- 待办任务
- 用户画像
- 产品战略
- 优先级排序
- 路线图
- 产品需求文档
- 用户故事
- 验收标准
- AI 产品管理
- 代理工作流
- 利益相关者对齐

这些是产品管理辅导与产物生成工作流，而非软件工程工作流。

## 提交与拉取请求指南

- 提交信息使用祈使句，并有清晰主题
- 拉取请求应包括简短摘要、相关议题、技能类型
- 如果新增技能，也要更新 `README.md` 中的目录、计数和表格

## 发布检查清单

- 更新 `README.md` 中的技能数量与表格
- 确认新技能已链接到正确分类
- 抽查 `README.md` 与 `CONTRIBUTING.md` 的交叉链接
- 若有重命名，确保所有路径和引用都同步更新
- 如果 Streamlit 测试版有变化，同时更新 `app/STREAMLIT_INTERFACE.md`、`app/.env.example` 以及发布说明

## 技能质量要求

- **教学优先**
- 代理就绪、自包含、实用
- 至少有一个具体示例，并展示推理过程
- 至少有一个明确的反模式，这不是装饰
- 术语首次出现时要解释清楚
- 不要以“更简洁”为理由删除解释、反模式或示例
