# 基于用户反馈的 Skills 改进

**日期：** 2025-11-28  
**状态：** 草稿  
**来源：** 两个在真实开发场景中使用 superpowers 的 Claude 实例

---

## 执行摘要

两次真实开发会话暴露出一些系统性问题。重点不是“个别实现失误”，而是当前 skills 还存在可预防但未拦住的失败模式。

主要主题：

1. 验证只看“操作成功”，没看“结果是否符合预期”
2. 后台进程会在多个 subagent 之间累积并互相干扰
3. 子代理收到的上下文过重
4. 交接前缺少自我复盘
5. mock 可能悄悄偏离真实接口
6. skill 虽然存在，但没有被读到或用到

## 识别出的核心问题

### 问题 1：配置变更验证缺口

现象：

- 改了配置
- 请求返回 200
- 就宣称改动生效

但真实结果可能仍然跑在旧配置上，例如切换了 provider，却仍返回旧模型。

根因：
`verification-before-completion` 更关注“命令成功”，没强制验证“可观察结果确实变化”。

影响：
高。会带来错误信心，让配置 bug 流入生产。

### 问题 2：后台进程累积

现象：

- 多个 subagent 先后启动服务
- 旧进程未清理
- 端口被占用或命中错误服务
- E2E 结果混乱

根因：
subagent 无状态，不知道前一个代理起过哪些进程。

影响：
中高。会造成假阳性、假阴性和调试混乱。

### 问题 3：子代理上下文膨胀

现象：

- 给子代理整份 plan
- 和只给 task、pattern、文件、验证命令相比，更慢、更散

根因：
无关内容占用了 token 和注意力。

影响：
中。执行更慢，失败重试更多。

### 问题 4：交接前没有自我复盘

现象：

- 实现者做完就汇报
- 没停下来重新审视自己的结果
- 本可自行发现的问题被留给 reviewer

影响：
中。增加 review 负担和来回沟通。

### 问题 5：Mock 和接口漂移

现象：

- 接口定义是 `close()`
- 实现错误调用 `cleanup()`
- mock 也跟着写成 `cleanup()`
- 测试通过，运行时崩溃

根因：
mock 是跟着错误实现写的，不是按接口定义写的。

影响：
高。测试给出虚假安全感。

### 问题 6：Reviewer 没先读文件

现象：

- code reviewer 子代理直接开始审
- 甚至报告“文件不存在”
- 实际上文件就在仓库里

根因：
review prompt 没明确要求先读文件。

影响：
中低。review 失败或不完整。

### 问题 7：修复流转多一跳

现象：

- 实现者在自检中已发现 bug
- 也知道怎么修
- 但流程仍要求先汇报，再派 fixer，再验证

影响：
低。主要是浪费时间。

### 问题 8：Skill 没被读到

现象：

- `testing-anti-patterns` 已存在
- 但写测试前没人去读

根因：
流程里没有强制或显式提示。

影响：
中。skill 投入被浪费。

## 建议改进

### 1. 强化 `verification-before-completion`

新增“配置变更验证”章节，强制检查：

1. 这次改动后，应该出现什么可观察差异
2. 这个差异在哪里能看到
3. 用什么命令看到它
4. 输出里是否真的出现了预期差异

重点：
不能只验证命令成功，必须验证结果真的变了。

### 2. 给 `subagent-driven-development` 增加流程卫生

针对会启动服务的任务：

- 启动前先清旧进程
- 确认端口已释放
- 结束后清理自己起的进程

### 3. 给 `subagent-driven-development` 增加精简上下文模式

拆成两种投喂方式：

- 完整计划模式：复杂、有关联依赖的任务
- 精简上下文模式：独立、模式清晰、照现有样例即可完成的任务

### 4. 给 `subagent-driven-development` 增加自我复盘步骤

交接前强制停一下，问自己：

- 这真的满足任务要求吗
- 有遗漏的边界吗
- 模式用对了吗
- 如果测试失败，根因到底是什么
- 还有什么能更好

### 5. 强化 `requesting-code-review`

在 reviewer prompt 开头显式写明：

- 先读哪些文件
- 读完再开始评审

### 6. 扩展 `testing-anti-patterns`

新增一节专门覆盖 “Mock-Interface Drift”：

- 先找接口定义
- 只 mock 接口真实存在的方法
- 不要按错误实现反推 mock

### 7. 测试任务前强制先读相关 skill

当任务涉及写测试时，在 prompt 里显式加入：

- 先读取 `superpowers:testing-anti-patterns`

### 8. 允许实现者先修掉自检发现的问题

如果自我复盘已经定位到可修复问题：

1. 先修
2. 重跑验证
3. 再汇报

## 实施顺序

### 阶段 1：高影响、低风险，先做

1. `verification-before-completion` 增加配置变更验证
2. `testing-anti-patterns` 增加 Mock-Interface Drift
3. `requesting-code-review` 增加强制读文件

### 阶段 2：中等改动，谨慎测试

4. `subagent-driven-development` 增加流程卫生
5. `subagent-driven-development` 增加自我复盘
6. `subagent-driven-development` 强制测试前先读 skill

### 阶段 3：先验证再决定

7. `subagent-driven-development` 增加精简上下文模式
8. 允许实现者直接修复自检发现的问题

## 开放问题

1. 精简上下文是否该默认启用
2. 自我复盘会不会拖慢简单任务
3. 流程卫生是否应抽成独立 skill
4. 强制读 skill 会不会让 prompt 过重

## 成功指标

1. 不再出现“配置没切过去但测试显示通过”
2. E2E 任务中明显减少端口冲突和命中旧服务
3. 不再出现 mock 跟着错误实现漂移
4. 实现者汇报中能看到自我复盘痕迹
5. 子代理开始主动引用相关 skill 规则

## 风险与缓解

### 风险：prompt 变重

缓解：

- 分阶段落地
- 只在相关场景启用相关约束
- 为任务类型提供不同模板

### 风险：分析瘫痪

缓解：

- gate function 保持轻量
- 精简上下文作为可选项
- 持续观察执行速度

### 风险：虚假安全感

缓解：

- 明确 skill 只降低常见失败概率
- 不能替代判断力

### 风险：技能间建议冲突

缓解：

- 跨 skill 做一致性检查
- 在 integration 章节说明关系
- 先用真实场景验证

## 建议

立刻推进阶段 1。  
阶段 2 先和实际使用者验证成本。  
阶段 3 暂缓，等更多真实反馈后再决定。
