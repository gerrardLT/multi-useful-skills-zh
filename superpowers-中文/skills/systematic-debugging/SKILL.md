# 系统化调试

## 概览

随机修修补补会浪费时间，还会制造新的 bug。快速补丁只会把底层问题盖住。

**核心原则：** 在尝试修复之前，永远先找到根因。只修症状就等于失败。

**违背这套流程的字面要求，也是在违背调试的精神。**

## 铁律

```text
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

如果你还没有完成第 1 阶段，就不能提出修复方案。

## 何时使用

适用于 **任何** 技术问题：
- 测试失败
- 生产环境 bug
- 异常行为
- 性能问题
- 构建失败
- 集成问题

**尤其要在以下场景使用：**
- 时间压力很大时（紧急情况最容易让人猜）
- “先来个 quick fix” 看起来特别诱人时
- 你已经试过多个修复方案时
- 上一个修复没生效时
- 你其实并没有完全理解这个问题时

**以下情况也绝不能跳过：**
- 问题看起来很简单（简单 bug 也有根因）
- 你很赶时间（赶时间只会保证返工）
- 经理要求“现在立刻修好”（系统化比乱试更快）

## 四个阶段

你必须按顺序完成每个阶段，才能进入下一阶段。

### 第 1 阶段：根因调查

**在尝试任何修复之前：**

1. **认真阅读错误信息**
   - 不要跳过 errors 或 warnings
   - 它们经常已经包含了准确线索
   - 把 stack trace 完整读完
   - 记下行号、文件路径和错误码

2. **稳定复现**
   - 你能稳定触发它吗？
   - 精确步骤是什么？
   - 每次都会发生吗？
   - 如果不能稳定复现 -> 继续收集数据，不要猜

3. **检查最近变更**
   - 最近改了什么可能导致这个问题？
   - 看 git diff、最近提交
   - 新依赖、配置变化
   - 环境差异

4. **在多组件系统里收集证据**

   **当系统由多个组件组成时（CI -> build -> signing，或 API -> service -> database）：**

   **在提出修复方案前，先加入诊断埋点：**
   ```text
   对每一个组件边界：
     - 记录进入该组件的数据
     - 记录离开该组件的数据
     - 验证环境/配置是否正确传递
     - 检查每一层的状态

   先跑一次，收集证据，确定到底是哪里坏了
   然后再分析证据，识别失效组件
   再去调查那个具体组件
   ```

   **示例（多层系统）：**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这能揭示：** 到底是哪一层失效（secrets -> workflow、workflow -> build 等等）。

5. **追踪数据流**

   **当错误出现在调用栈很深的地方时：**

   参见当前目录中的 `root-cause-tracing.md`，里面有完整的反向追踪技巧。

   **快速版本：**
   - 错值最早从哪里来的？
   - 是谁把这个坏值传给当前代码的？
   - 一直往上追，直到找到真正源头
   - 在源头修，而不是在症状点修

### 第 2 阶段：模式分析

**先识别模式，再动手修：**

1. **寻找正常工作的例子**
   - 在同一个代码库里找相似且正常的实现
   - 有什么是“和它类似，但能正常工作”的？

2. **对照参考实现**
   - 如果你在实现某种 pattern，就把参考实现 **完整读完**
   - 不要只扫一眼，要逐行读
   - 在应用前，先完整理解这个模式

3. **识别差异**
   - 正常版本和出问题版本有什么不同？
   - 每个差异都列出来，再小也不要忽略
   - 不要想当然地说“这个差异不重要”

4. **理解依赖关系**
   - 它依赖了哪些其他组件？
   - 依赖哪些设置、配置和环境？
   - 它基于哪些前提假设？

### 第 3 阶段：假设与验证

**按科学方法来：**

1. **形成单一假设**
   - 明确说出来：`"I think X is the root cause because Y"`
   - 把它写下来
   - 要具体，不要模糊

2. **最小化验证**
   - 只做能验证假设的 **最小改动**
   - 一次只改一个变量
   - 不要一次修多件事

3. **验证之后再继续**
   - 成功了吗？成功 -> 进入第 4 阶段
   - 没成功？形成一个 **新的** 假设
   - 不要在原修复上继续叠更多“顺手一改”

4. **当你不知道时**
   - 直接说 `"I don't understand X"`
   - 不要装懂
   - 请求帮助
   - 继续研究

### 第 4 阶段：实现

**修根因，不修症状：**

1. **创建失败测试用例**
   - 先写出最简单的复现场景
   - 能自动化就自动化
   - 没框架也要写一次性测试脚本
   - **修之前必须先有**
   - 正确写 failing test 时请使用 `superpowers:test-driven-development` skill

2. **只实现一个修复**
   - 只针对已确认的根因下手
   - 一次只改一个点
   - 不要“既然来了顺便优化一下”
   - 不要把重构打包进来

3. **验证修复**
   - 测试现在通过了吗？
   - 有没有其他测试被你搞坏？
   - 问题真的解决了吗？

4. **如果修复没成功**
   - 停下
   - 数一数：你已经试了几次修复？
   - 如果 < 3：回到第 1 阶段，用新信息重新分析
   - **如果 >= 3：停止，并开始质疑架构（见下面第 5 步）**
   - 在没有做架构讨论前，不要去尝试第 4 次修复

5. **如果 3 次以上修复都失败：开始质疑架构**

   **这通常意味着架构层面有问题：**
   - 每修一次，就在另一个地方暴露出新的共享状态 / 耦合 / 问题
   - 修复开始需要“大规模重构”才能落地
   - 每次修一个点，都会在别处引入新症状

   **这时要停下来质疑基本面：**
   - 这个模式本身是不是从根上就不对？
   - 我们是不是只是因为惯性才一直硬撑着不改？
   - 现在该做的是重构架构，而不是继续补症状吗？

   **在尝试更多修复前，先和你的人类搭档讨论。**

   这已经不是“假设失败”，而是“架构错了”。

## 红旗信号：立刻停下并回到流程

如果你发现自己在想这些话：
- “先 quick fix 一下，之后再查”
- “先改一下 X 试试能不能好”
- “多改几个地方再跑测试”
- “先别写测试了，我手动验证一下”
- “大概就是 X，让我先修掉”
- “我没完全懂，但这个可能能行”
- “pattern 是这么写的，但我想稍微改着用”
- “主要问题有这些：[直接列修复方案，却没调查]”
- 还没追踪数据流就开始提解决方案
- **“再试最后一次修复”**（但你已经试了 2 次以上）
- **每修一次都会在另一个地方冒出新问题**

**以上任意一种都意味着：停下，回到第 1 阶段。**

**如果已经失败 3 次以上：** 开始质疑架构（见第 4.5 节）

## 你的人类搭档发出的“你做错了”的信号

**注意这些重定向：**
- `"Is that not happening?"` - 你在没有验证的前提下做了假设
- `"Will it show us...?"` - 你本来应该先补证据收集
- `"Stop guessing"` - 你在没理解问题前就开始提修复
- `"Ultrathink this"` - 该质疑基本面，而不只是处理表面症状
- `"We're stuck?"`（带着挫败感）- 说明你的方法已经不工作了

**一旦听到这些：停下，回到第 1 阶段。**

## 常见自我合理化

| Excuse | Reality |
|--------|---------|
| “问题很简单，不需要流程” | 简单问题也有根因。这个流程对简单 bug 也很快。 |
| “这是紧急情况，没时间走流程” | 系统化调试比乱试乱撞更快。 |
| “先试这个，再调查也不迟” | 第一次修复就决定了后面的模式。一开始就做对。 |
| “我先确认修复有效，再补测试” | 没测试的修复站不住。先写测试才能证明。 |
| “一次修多个地方更省时间” | 你没法隔离到底是什么生效了，还会引入新 bug。 |
| “参考实现太长了，我自己改着套” | 只理解一半，几乎注定出 bug。完整读完。 |
| “我看出来问题了，直接改吧” | 看到症状 != 理解根因。 |
| “再试最后一次修复” | 3 次以上失败 = 架构问题。该质疑模式，而不是继续补。 |

## 快速参考

| Phase | Key Activities | Success Criteria |
|-------|---------------|------------------|
| **1. Root Cause** | 读错误、复现、看变更、收集证据 | 理解 WHAT 和 WHY |
| **2. Pattern** | 找正常例子、对比实现 | 识别关键差异 |
| **3. Hypothesis** | 建立理论、做最小验证 | 假设被确认或被替换 |
| **4. Implementation** | 写测试、修复、验证 | 问题解决，测试通过 |

## 当流程最终得出“没有明确根因”时

如果系统化调查后发现，问题确实是环境因素、时序因素或外部系统导致的：

1. 你依然已经正确完成了整个流程
2. 把你调查过的内容记录下来
3. 实现合适的处理方式（如 retry、timeout、错误提示）
4. 增加监控或日志，方便以后继续调查

**但是：** 95% 所谓“没有根因”的情况，其实只是调查还不够完整。

## 真实世界影响

来自真实调试会话：
- 系统化方法：15-30 分钟解决
- 随机乱试方法：2-3 小时空转
- 一次修对率：95% vs 40%
- 引入新 bug 的概率：几乎为零 vs 非常常见
