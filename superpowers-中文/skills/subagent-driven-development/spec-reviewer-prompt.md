# Spec 一致性审查 Prompt 模板

在派发 `spec` 一致性审查子代理时使用这份模板。

**目的：** 验证 implementer 做出来的内容，是否严格等于要求的内容，不多也不少。

```text
Task tool (general-purpose):
  description: "审查任务 N 的 spec 一致性"
  prompt: |
    你正在审查某个实现是否符合它的 specification。

    ## 原始要求

    [任务需求完整原文]

    ## Implementer 声称自己实现了什么

    [摘自 implementer 的回报]

    ## 关键要求：不要相信这份回报

    implementer 完成得很快，这份回报可能并不完整，也可能不准确，甚至可能过于乐观。
    你**必须**独立核验所有内容。

    **不要：**
    - 直接相信他们说自己做了什么
    - 相信他们对完整性的自我判断
    - 接受他们对需求的个人解释

    **要做：**
    - 读取他们实际写出来的代码
    - 逐条对照需求与真实实现
    - 检查是否有他们声称做了但其实没做的部分
    - 检查是否偷偷做了他们没提到的额外内容

    ## 你的工作

    读取实现代码，并验证：

    **缺失要求：**
    - 他们是否实现了所有被要求的内容？
    - 有没有跳过或漏掉的要求？
    - 有没有声称“已完成”，但实际没真正实现？

    **额外或不必要的工作：**
    - 他们是否做了需求里没要求的内容？
    - 是否过度设计，或加入了不必要的特性？
    - 是否加了 spec 里根本没有的“顺手优化”？

    **理解偏差：**
    - 他们是否误解了需求？
    - 是否解决了错误的问题？
    - 是否做对了功能，但做法偏离要求？

    **以读代码为准，不要信回报。**

    输出格式：
    - Spec compliant（如果你在读完代码后确认全部匹配）
    - Issues found: [具体列出缺失或额外内容，并附 file:line 引用]
```

## 中文说明

这位 reviewer 的职责不是看“代码写得好不好”，而是看“它有没有严格按 spec 来做”。

重点：

- 先默认 implementer 的汇报不可靠
- 必须独立读代码验证
- 重点找三类问题：
  - 少做了什么
  - 多做了什么
  - 理解偏了什么

只有这一关通过之后，才进入代码质量评审。
