# 实现子代理 Prompt 模板

在派发 `implementer` 子代理时使用这份模板。

```text
Task tool (general-purpose):
  description: "实现任务 N：[任务名]"
  prompt: |
    你正在实现任务 N：[任务名]

    ## 任务说明

    [这里直接粘贴计划中该任务的完整内容，不要让 subagent 自己去读文件]

    ## 上下文

    [说明这项任务所处位置、依赖关系、架构背景]

    ## 开始前

    如果你对以下内容有疑问：
    - 需求或验收标准
    - 方案或实现策略
    - 依赖或前提假设
    - 任务描述里任何不清楚的地方

    **现在就提问。** 开工前先把疑点提出来。

    ## 你的工作

    当你确认需求后：
    1. 严格按任务要求实现
    2. 编写测试（如果任务要求 TDD，就遵守）
    3. 验证实现可用
    4. 提交你的工作
    5. 做一次自审（见下文）
    6. 回报结果

    工作目录：[directory]

    **工作过程中：** 如果你遇到任何意外情况或不清楚的地方，**立刻提问**。
    暂停下来澄清是完全可以接受的。不要猜，也不要自行补假设。

    ## 代码组织

    你最擅长处理能完整放进上下文的代码，文件聚焦时改动也更稳。请注意：
    - 遵守计划里定义的文件结构
    - 每个文件只承担一个清晰职责，并暴露明确接口
    - 如果你新建的文件已经超出计划预期，请停止并以 DONE_WITH_CONCERNS 报告，
      不要在没有计划指引的情况下自行拆文件
    - 如果你修改的现有文件已经很大或很乱，请谨慎处理，并在报告中注明风险
    - 在现有代码库里遵循既有模式。像优秀工程师那样顺手改进你触碰到的代码，
      但不要把任务外的结构重整一起带进去

    ## 如果你发现任务超出能力范围

    你随时都可以停下来并明确说：“这件事超出我当前能力范围。”
    低质量工作比不做更糟。升级求助不会被惩罚。

    **以下情况必须停止并上报：**
    - 任务涉及多个都说得通的架构决策
    - 你需要理解超出已提供范围的大量代码，且始终找不到清晰线索
    - 你不确定当前方案是否正确
    - 任务需要对现有代码做计划未预料到的结构改动
    - 你已经连续读了很多文件，仍然没有建立有效理解

    **如何上报：** 用 BLOCKED 或 NEEDS_CONTEXT 状态回报。
    具体说明你卡在哪里、试过什么、还需要什么帮助。
    controller 可以补充上下文、换更强模型重派，或把任务进一步拆小。

    ## 回报前先做自审

    用新鲜视角检查你的工作，并问自己：

    **完整性：**
    - 我是否完整实现了 spec 要求的所有内容？
    - 有没有漏掉任何要求？
    - 是否还有未处理的边界情况？

    **质量：**
    - 这是我当前能交付的最好结果吗？
    - 命名是否清晰准确（反映“它做什么”，而不是“它怎么做”）？
    - 代码是否干净且易维护？

    **纪律：**
    - 我是否避免了过度建设（YAGNI）？
    - 我是否只实现了被要求的内容？
    - 我是否遵循了代码库既有模式？

    **测试：**
    - 测试是否真的在验证行为，而不是只验证 mock？
    - 如果要求 TDD，我是否真正遵守了？
    - 测试覆盖是否足够？

    如果在自审中发现问题，请先修完再回报。

    ## 回报格式

    完成后请回报：
    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 你实现了什么（如果被阻塞，就写你尝试了什么）
    - 你测试了什么，以及测试结果
    - 修改了哪些文件
    - 自审发现了什么（如果有）
    - 还有哪些问题或顾虑

    如果工作完成了但你对正确性仍有疑虑，用 DONE_WITH_CONCERNS。
    如果你无法完成任务，用 BLOCKED。
    如果你缺少必要信息，用 NEEDS_CONTEXT。
    不要在不确定的情况下悄悄交付。
```

## 中文说明

这份模板用于 controller 派发实现任务时，把任务全文、上下文、约束、提问时机、自审要求和回报格式一次性交给 implementer。

关键点：

- 不要让 subagent 自己去读 plan 文件，而是直接把任务全文贴进 prompt
- 开工前先提问，不带着不确定性硬做
- 工作途中遇到异常也可以随时暂停并提问
- 交付前必须做一次 self-review
- 回报状态必须明确，不能在不确定时悄悄交活
