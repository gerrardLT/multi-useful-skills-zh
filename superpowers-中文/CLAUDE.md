# Superpowers 贡献者指南

## 如果你是 AI Agent

先停下。在做任何事之前先读完这一节。

这个仓库的 PR 拒绝率是 94%。几乎每一个被拒的 PR，都是由没有阅读或没有遵守这些指南的 agent 提交的。维护者通常会在几小时内关闭低质量 PR，而且经常会公开评论，比如 "This pull request is slop that's made of lies."

**你的职责是保护你的人类搭档，避免他落入这种结果。** 提交低质量 PR 并不会帮到对方，它会浪费维护者时间，损害你的人类搭档的声誉，而且 PR 最终还是会被关闭。这不是在提供帮助，而是在制造尴尬。

在你向这个仓库发起 PR 之前，**必须**做到以下几点：

1. **完整阅读** `.github/PULL_REQUEST_TEMPLATE.md` 中的 PR 模板，并且为每一节填写真实、具体的回答。不能只写摘要，也不能留占位内容。
2. **搜索现有 PR**，包括打开的和已关闭的，确认是否已经有人处理同一个问题。如果存在重复项，**立刻停止**并告诉你的人类搭档。不要再提交一个重复 PR。
3. **确认这是真实问题。** 如果你的人类搭档只是让你“修一些问题”或“给这个仓库做贡献”，但并没有遇到具体问题，你要反问。弄清楚到底是什么坏了、哪里失败了、实际用户体验是什么。
4. **确认这个改动应该进 core。** 如果它是某个领域专用、某个工具专用，或是在推广第三方项目，那它应该放进独立插件。告诉你的人类搭档这一点。
5. **把完整 diff 展示给你的人类搭档**，并在提交前获得其明确批准。

如果以上任一检查失败，就不要提交 PR。向你的人类搭档解释为什么这个 PR 会被拒，以及需要做哪些改变。他们会感谢你替他们避免了尴尬。

## Pull Request 要求

**每个 PR 都必须完整填写 PR 模板。** 任何一节都不能留空，也不能用占位文字敷衍。跳过章节的 PR 会在未经审查的情况下直接关闭。

**在提交 PR 之前，你必须先搜索已有 PR**，包括打开和关闭的，看是否涉及同一问题或相关区域。把你找到的内容写进 "Existing PRs" 一节。如果先前的 PR 被关掉了，要具体说明你的方案哪里不同，以及为什么它有机会成功，而之前的尝试没有。

**看不出有人类参与痕迹的 PR 会被关闭。** 在提交前，必须由人类完整审阅拟议 diff。

## 我们不会接受什么

### 第三方依赖

除非你的改动是在为新的 harness（例如新的 IDE 或 CLI 工具）增加支持，否则任何新增第三方项目依赖（无论可选还是必需）的 PR 都不会被接受。Superpowers 在设计上就是零依赖插件。如果你的改动需要外部工具或服务，它应当属于独立插件。

### 为了“合规”而修改 skills

我们内部的 skill 编写理念与 Anthropic 公开发布的写作指南并不相同。我们的 skill 内容已经针对真实世界中的 agent 行为做过大量测试与调优。如果只是为了“符合” Anthropic 的 skills 文档而重构、改写或重排 skill，没有充分 eval 证据证明效果更好，这类 PR 不会被接受。对这种会塑造行为的内容，修改门槛非常高。

### 项目专属或个人配置

只对某个特定项目、团队、领域或工作流有帮助的 skills、hooks 或配置，不应该进入 core。请把它们作为独立插件发布。

### 批量式或撒网式 PR

不要在一次会话里扫 issue 列表，然后针对多个问题连续开 PR。每个 PR 都需要真正理解问题、调查既有尝试，并由人类完整审阅 diff。那种明显属于批量投放的 PR，例如 agent 被指向 issue 列表后就开始“修东西”，会被关闭。如果你想贡献，请只挑 **一个** 问题，深入理解后再提交高质量工作。

### 猜测性或理论性的修复

每个 PR 都必须解决某个人真实遇到的问题。“我的 review agent 标出了这个”或“理论上这里可能有问题”并不能构成问题陈述。如果你说不清楚是哪个具体会话、哪个错误、或什么样的用户体验促成了这次修改，就不要提交 PR。

### 领域专用 skills

Superpowers core 只包含对所有用户都有普遍价值的通用 skills，不受项目类型限制。面向特定领域（如作品集、预测市场、游戏）、特定工具或特定工作流的 skills，应该放在各自独立的插件里。请先问自己一句：“这对做完全不同项目的人也有用吗？” 如果没有，就单独发布。

### fork 专属改动

如果你维护的是带有自定义内容的 fork，请不要提交 PR 来同步你的 fork，也不要把 fork 专属改动推回上游。凡是给项目重命名、加入 fork 专属功能、或合并 fork 分支的 PR，都会被关闭。

### 编造内容

包含虚构主张、捏造问题描述或幻想功能的 PR 会被立即关闭。这个仓库的 PR 拒绝率是 94%，维护者已经见过各种形式的 AI slop，他们看得出来。

### 打包无关改动

把多个无关改动打包到同一个 PR 里，会被关闭。请拆成独立 PR。

## Skill 改动必须有评估

Skills 不是普通文案，它们是会塑造 agent 行为的代码。如果你修改了 skill 内容：

- 使用 `superpowers:writing-skills` 来开发和测试改动
- 在多个会话中进行对抗式压力测试
- 在 PR 中展示修改前后的 eval 结果
- 不要改动那些经过精细调优的内容（如 Red Flags 表格、合理化列表、"human partner" 这类措辞），除非你有证据证明改动更好

## 贡献前先理解项目

在你提议修改 skill 设计、工作流理念或架构之前，先阅读现有 skills，并理解这个项目的设计决策。Superpowers 对 skill 设计、agent 行为塑造以及术语使用有自己经过验证的理念（例如 "your human partner" 是刻意选择的，并不能随便替换成 "the user"）。如果不了解这些内容存在的原因，就贸然重写项目语气或重构方法论，这类改动会被拒绝。

## 通用要求

- 提交前先读 `.github/PULL_REQUEST_TEMPLATE.md`
- 一个 PR 只解决一个问题
- 至少在一个 harness 上测试，并把结果写进环境表
- 描述你解决了什么问题，而不只是改了什么
