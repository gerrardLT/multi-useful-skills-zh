# Extract Flow

识别可复用的模式、组件和设计 token，然后把它们提取并整合进设计系统，以便系统化复用。

## 第 1 步：找到设计系统

先找到设计系统、组件库，或者共享 UI 目录。理解它的结构：组件是如何组织的、命名约定是什么、设计 token 的结构、import / export 方式分别怎样。

**关键：** 如果当前根本没有设计系统，不要直接拍脑袋新建。优先使用 Codex 的结构化提问工具；如果当前环境没有这个工具，就在对话里直接发问，先确认你无法从代码里推断出的信息。一定要先弄清楚对方偏好的存放位置和结构。

## 第 2 步：识别可提取模式

在目标区域中寻找提取机会：

- **重复组件**：相似的 UI 模式出现了 3 次以上（如按钮、卡片、输入框）
- **硬编码数值**：颜色、间距、字体、阴影这类本该沉淀成 token 的值
- **不一致变体**：同一个概念被做出了多个版本
- **组合模式**：会重复出现的布局或交互结构（表单行、工具栏分组、空状态）
- **文字样式**：重复出现的字号 + 字重 + 行高组合
- **动画模式**：重复出现的 easing、duration 或 keyframe 组合

先判断价值：只有在相同意图下被使用 3 次以上的内容，才值得提取。过早抽象通常比重复更糟。

## 第 3 步：规划提取方案

建立一份系统化计划：

- **要提取哪些组件**：哪些 UI 元素应该成为可复用组件？
- **要创建哪些 token**：哪些硬编码值应该转成设计 token？
- **需要支持哪些变体**：每个组件需要涵盖哪些差异？
- **命名规则**：组件名、token 名、prop 名都应和现有模式保持一致
- **迁移路径**：怎样把现有使用点逐步重构到新的共享版本上

**重要：** 设计系统应当渐进式生长。只提取那些“现在已经明显可复用”的东西，不要为未来可能的场景提前建一整套。

## 第 4 步：提取并增强

构建经过改良、可复用的版本：

- **组件**：清晰的 props API、合理默认值、适合不同场景的变体、内建无障碍支持（ARIA、键盘导航、焦点管理）、文档和使用示例
- **设计 token**：命名清楚（区分 primitive 和 semantic）、层级结构合理、说明何时该使用哪个 token
- **模式**：说明什么情况下用、给出代码示例，并描述变体与组合方式

## 第 5 步：迁移

用新的共享版本替换现有用法：

- **找出所有实例**：搜索你刚提取的那些模式
- **系统化替换**：逐个更新使用点，改为依赖共享版本
- **充分测试**：确认视觉和功能都保持等价
- **删除死代码**：把旧实现移除掉

## 第 6 步：写文档

更新设计系统文档：

- 把新组件加入组件库
- 记录 token 的用法与取值
- 添加示例与使用准则
- 更新 Storybook 或组件目录

**绝对不要：**

- 提取那些一次性的、强上下文绑定的实现，而不先做合理抽象
- 造出“什么都能干、结果什么都不好用”的超级泛化组件
- 不看现有设计系统约定就直接提取
- 跳过 TypeScript 类型或 prop 文档
- 为每一个孤立数值都建 token（token 应当有语义）
- 把意图不同、只是长得像的东西强行提成一个组件

记住：好的设计系统是一个活的系统。模式出现时再提取，提取时做适度增强，并长期一致地维护它。
