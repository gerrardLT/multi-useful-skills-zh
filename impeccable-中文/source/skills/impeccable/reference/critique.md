> **还需要额外上下文**：这个界面到底要完成什么目标。

### 收集评估

启动两个彼此独立的评估。**两者都不能看到对方的输出**——正是这种隔离，才能让最终的综合评分保持诚实。如果在同一个思路里静默做完两项，它们就会互相锚定；不要因为成本、速度或上下文长度而省略这个隔离。

把每个评估都委托给一个独立的 sub-agent（Claude Code 的 `Agent` 工具、Codex 的 subagent spawning 等）。每个 sub-agent 都以文本形式返回结构化发现。此时不要把发现直接输出给用户。

只有在当前环境确实无法创建 sub-agents 时，才退回到单线程、同一个脑内顺序执行。

**标签页隔离**：如果当前有浏览器自动化能力，那么每个评估**都必须**打开自己的新标签页。即便当前已经有同一个 URL 的正确页面打开，也绝不能复用。这样能防止两个评估互相污染页面状态。

#### 评估 A：LLM 设计评审

读取相关源文件（HTML、CSS、JS / TS）；如果有浏览器自动化能力，还要对 live 页面做视觉检查。**为这个评估创建一个新标签页**，不要复用已有页面。导航完成后，用下面的方式设置 document title 给标签页打标：
```javascript
document.title = '[LLM] ' + document.title;
```
以设计总监的视角思考。评估：

**AI 垃圾检测（关键）**：这看起来像不像其他所有 AI 生成的界面？对照父级 impeccable skill 中所有 **不要** 指南（当前上下文已加载）。检查是否存在 AI 常见配色、渐变文字、深色发光、玻璃拟态、hero 指标布局、一模一样的卡片网格、泛化字体，以及其他所有明显痕迹。**测试标准**：如果有人说 “AI 做的”，你会不会立刻相信？

**整体设计评审**：视觉层级（视线路径、主行动清晰度）、信息架构（结构、分组、认知负荷）、情绪共鸣（是否匹配品牌和受众？）、可发现性（交互元素是否一眼可见？）、构图（平衡、留白、节奏）、排版（层级、可读性、字体选择）、颜色（是否有目的、是否统一、是否可访问）、状态与边界情况（空、加载、错误、成功）、微文案（清晰度、语气、帮助性）。

**认知负荷**（参考 [cognitive-load](cognitive-load.md)）：
- 执行 8 项认知负荷检查清单。报告失败项数量：0-1 = 低（好），2-3 = 中等，4+ = 严重。
- 统计每个决策点上用户可见的选项数。如果超过 4 个，就标记。
- 检查是否使用了渐进式披露：复杂度是否只在真正需要时才暴露？

**情绪旅程**：
- 这个界面唤起了什么情绪？这是有意设计的吗？
- **峰终定律**：最强烈的时刻是否是正向的？整个体验结尾是否收得好？
- **情绪低谷**：在高风险时刻（支付、删除、提交）是否有焦虑峰值？是否有设计干预（进度指示、安抚文案、撤销选项）？

**尼尔森启发式**（参考 [heuristics-scoring](heuristics-scoring.md)）：
对 10 条启发式逐条打 0-4 分。这份评分稍后会在报告里展示。

返回的结构化结果应覆盖：AI 垃圾判定、启发式评分、认知负荷评估、做得好的地方（2-3 条）、优先问题（3-5 条，包含 问题 / 原因 / 修复）、次级观察，以及有刺激性的提问。

#### 评估 B：自动化检测

运行内置的确定性检测器。它会标出 25 种具体模式（AI 垃圾痕迹 + 通用设计质量问题）。

**CLI 扫描**：
```bash
npx impeccable --json [--fast] [target]
```

- 把 HTML / JSX / TSX / Vue / Svelte 文件或目录作为 `[target]`（任何带标记的内容）。不要把纯 CSS 文件作为 target。
- 如果 target 是 URL，就跳过 CLI 扫描（它需要 Puppeteer）。这种情况只用浏览器可视化。
- 如果目录很大（200+ 可扫描文件），加 `--fast`（纯正则，跳过 jsdom）
- 如果超过 500 个文件，就缩小范围，或询问用户
- 退出码 0 = 干净，2 = 有发现

**浏览器可视化**：当存在浏览器自动化能力，且 target 是一个可在浏览器查看的页面时，这一步是**必需的**。`[Human]` overlay 标签页是交给用户的可视化结果；没有它，这次评审就不完整。只有当 target 根本不是可视化页面（纯 CSS 文件、非浏览器对象）时，才可以跳过。

overlay 是一个 **给用户看的视觉辅助工具**。它会直接在浏览器里高亮问题。不要靠滚动页面截图来展示这些 overlays，而应通过读取 console 输出，以编程方式拿到结果。

1.  **启动 live detection server**：
    ```bash
    npx impeccable live &
    ```
    记下 stdout 打印出的端口（自动分配）。如果要固定端口，可加 `--port=PORT`。
2.  **创建一个新标签页**，并导航到页面（本地文件用 dev server URL，线上页面直接用目标 URL）。不要复用现有标签页。
3.  **通过 `javascript_tool` 给标签页打标**，方便用户区分：
    ```javascript
    document.title = '[Human] ' + document.title;
    ```
4.  **滚动到顶部**，确保注入前页面位于最顶端
5.  **通过 `javascript_tool` 注入**（把 PORT 替换成第 1 步中的实际端口）：
    ```javascript
    const s = document.createElement('script'); s.src = 'http://localhost:PORT/detect.js'; document.head.appendChild(s);
    ```
6.  等待 2-3 秒，让检测器渲染 overlays
7.  **通过 `read_console_messages` 读取结果**，pattern 使用 `impeccable`。检测器会把所有发现以前缀 `[impeccable]` 写到控制台。不要滚动页面去截 overlays。
8.  **清理**：结束 live server：
    ```bash
    npx impeccable live stop
    ```

对于多视图目标，在 3-5 个有代表性的页面上做注入。如果注入失败，就只保留 CLI 结果继续。

返回内容包括：CLI 发现（JSON）、浏览器 console 发现（如果有），以及发现的任何误报。

### 询问用户

**在呈现完发现之后**，基于你实际发现的问题，提出有针对性的问题。{{ask_instruction}} 用户的回答将决定接下来的行动方案。

问题要沿着下面这些方向，但必须根据实际发现改写，**不要问泛泛的问题**：

1.  **优先级方向**：基于发现的问题，问用户眼下最在意哪一类。比如：“我发现了视觉层级、颜色使用和信息过载的问题。我们应该先处理哪个领域？” 给出 2-3 个最主要的问题类别作为选项。
2.  **设计意图**：如果评审发现了语气偏差，就问用户这是不是故意的。比如：“界面感觉临床和企业化。这是预期的基调，还是应该感觉更温暖/更大胆/更活泼？” 提供 2-3 种能修正当前问题的语气方向作为选项。
3.  **范围**：问用户这次想解决到什么程度。比如：“我发现了 N 个问题。想要全部解决，还是只关注前 3 个？” 提供像 “仅前 3 个”、“所有问题”、“仅严重问题” 这样的范围选项。
4.  **约束**（可选，仅在相关时提）：如果发现涉及面很广，问用户是否有任何区域不能动。比如：“是否有任何部分应该保持原样？” 这样可以避免后续计划碰到用户认为已经定稿的区域。

**提问规则**：
- 每个问题都必须引用报告中的具体发现。绝不能问泛泛的 “你的受众是谁？” 之类问题。
- 总数控制在 2-4 个以内。尊重用户时间。
- 给出具体选项，而不是完全开放式提问。
- 如果发现很直接（例如只有 1-2 个明确问题），就跳过提问，直接进入推荐行动。

...

- 每一项描述都要带足够上下文，让这个命令知道该把注意力放在哪
- 每个优先问题都要映射到合适命令
