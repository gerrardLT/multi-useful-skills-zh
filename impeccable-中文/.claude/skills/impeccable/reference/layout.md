评估并改进那些显得单调、拥挤或结构薄弱的布局与间距，把泛化排列变成有意图、有节奏的构图。

---

## Register

Brand：偏向不对称构图、使用 `clamp()` 的流动间距，以及带有刻意越界感的强调式布局。节奏来自对比——紧密分组和大幅留白并置。

Product：偏向可预测的网格、一致的密度，以及熟悉的导航模式。响应式行为应体现在结构上（如收起侧边栏、响应式表格），而不是依赖流体排版。对 product 来说，一致性本身就是 affordance。

## 评估当前布局

先分析当前空间设计到底哪里弱：

1. **间距**：
   - 间距是统一的，还是随意拍脑袋的？（随机 padding / margin）
   - 是不是所有地方都一样大？（处处等间距 = 没有节奏）
   - 相关元素之间是否足够紧密，而不同组之间是否有足够慷慨的留白？

2. **视觉层级**：
   - 做一下 squint test：把你的（比喻意义上的）眼睛眯起来，是否还能一眼识别出最重要元素、第二重要元素，以及清楚的分组关系？
   - 当前层级实现有效吗？（很多时候只靠空间和字重就够了——但现在这套做法真的有效吗？）
   - 留白是否在引导视线去看真正重要的内容？

3. **网格与结构**：
   - 底层结构是否清晰，还是整个布局像随机摆放？
   - 是否到处都在使用一模一样的卡片网格？（icon + heading + text 的重复）
   - 是不是所有内容都被居中了？（左对齐结合适度不对称，往往更像经过设计，但这不是死规则）

4. **节奏与变化**：
   - 布局是否有视觉节奏？（紧与松之间有交替）
   - 每个 section 是否都长得一样？（会带来单调重复）
   - 是否存在一些刻意设计的惊喜或强调时刻？

5. **密度**：
   - 布局是否太挤？（缺少呼吸空间）
   - 布局是否太散？（没有目的的大片留白）
   - 密度是否匹配内容类型？（数据密集型 UI 应该更紧，营销页需要更多空气）

**CRITICAL**：很多界面之所以“哪里都没错，但就是不对”，根因往往是布局问题，而不是颜色或字体。空间本身就是设计材料，要有意图地使用它。

## 规划布局改进

参考 [spatial design reference](spatial-design.md) 获取更详细的网格、节奏与 container query 指导。

建立一套系统方案：

- **Spacing system**：使用一致的间距尺度——可以是框架自带尺度（如 Tailwind）、基于 rem 的 tokens，或自定义系统。具体数值没那么重要，重要的是一致性。
- **Hierarchy strategy**：空间要如何表达重要性？
- **Layout approach**：什么样的结构最适合当前内容？1D 用 Flex，2D 用 Grid，复杂页面可用命名区域。
- **Rhythm**：哪些地方要紧，哪些地方要松？

## 系统性地改进布局

### 建立间距系统

- 使用统一的间距尺度——框架尺度（Tailwind 等）、基于 rem 的 token，或自定义尺度都可以。关键在于所有值都来自一个定义好的集合，而不是任意数字。
- 如果使用自定义属性，请采用语义化命名：`--space-xs` 到 `--space-xl`，不要用 `--spacing-8`
- 同级元素之间的间距优先用 `gap`，而不是 margins——这样可以避开 margin collapse 这类补丁问题
- 使用 `clamp()` 让大屏上的间距更有呼吸感

### 建立视觉节奏

- **紧密分组** 用于强相关元素（兄弟元素之间 8-12px）
- **慷慨分隔** 用于不同 section（48-96px）
- **Section 内部也要有变化**——不是每一行都必须使用同一个 gap
- **不对称构图**——在合理的时候，打破“所有内容都规整居中”的可预测模式

### 选择正确的布局工具

- **1D 布局优先用 Flexbox**：项目列表、导航栏、按钮组、卡片内部，大多数组件内排布都属于这一类。Flex 更简单，也更适合绝大多数场景。
- **2D 布局再用 Grid**：页面级结构、仪表盘、数据密集型界面，以及那些确实需要同时控制行和列的地方。
- **不要默认用 Grid**，如果 Flexbox + `flex-wrap` 已经更简单、更灵活。
- 对响应式网格使用 `repeat(auto-fit, minmax(280px, 1fr))`，很多时候能避免手写断点。
- 对复杂页面结构使用命名 grid areas（`grid-template-areas`），并在断点上重定义。

### 打破卡片网格的单调感

- 不要默认“什么都放进卡片网格”——很多时候只靠间距和对齐就能形成清晰分组
- 只有在内容真的独立且具备操作性时才用卡片——绝不要卡片里再套卡片
- 通过改变卡片尺寸、跨列、或混合非卡片内容来打破重复

### 加强视觉层级

- 用最少的维度建立足够清晰的层级。有时只靠空间就够了——一个元素周围留出更大留白，本身就会吸引视线。很多真正精致的设计只靠空间和字重就能做出节奏。只有当这些还不够时，才增加颜色或字号对比。
- 理解阅读流——在 LTR 语言中，视线天然从左上扫到右下，但主要动作的落位还是取决于场景（例如对话框里常在右下，导航里则常在顶部）。
- 通过 proximity 与 separation 建立清楚的内容分组。

### 管理深度与层次

- 建立语义化的 z-index 尺度（dropdown -> sticky -> modal-backdrop -> modal -> toast -> tooltip）
- 建立统一的 shadow 尺度（sm -> md -> lg -> xl）——阴影应保持克制
- Elevation 用来强化层级，而不是装饰

### 光学校正

- 如果某个 icon 几何上居中，但视觉上看着偏了，就做微调——但前提是你真的确认它看起来不对。不要猜着调。

**NEVER**：
- 不要使用超出尺度系统的任意间距值
- 不要让所有间距都一样大——变化才会带来层级
- 不要把一切都包进卡片——不是所有内容都需要容器
- 不要卡片套卡片——内部层级应通过间距和分隔线解决
- 不要到处都是一模一样的卡片网格（icon + heading + text 的重复）
- 不要什么都居中——左对齐配合适度不对称通常更像经过设计
- 不要把 hero metric 布局（大数字、小标签、统计、渐变）当成默认模板。如果展示的是真实用户数据，醒目的指标可以成立；但必须是真数据，而不是装饰性数字。
- 不要在 Flex 就够用时默认上 CSS Grid——永远用最简单的工具解决问题
- 不要使用任意 z-index 值（999、9999）——建立语义尺度

## 验证布局改进

- **Squint test**：在视线模糊时，是否仍能识别 primary、secondary 和分组？
- **节奏**：页面是否具有紧与松交替的令人满意的节拍？
- **层级**：用户能否在 2 秒内看出最重要的内容？
- **呼吸感**：布局是否舒适，而不是拥挤或浪费？
- **一致性**：间距系统是否被统一应用？
- **响应性**：布局在不同屏幕尺寸下是否自然适配？

记住：空间是最被低估的设计工具。只要节奏和层级做对了，哪怕是简单内容，也会显得精致且有意图。

## Live 模式标志参数

每个 variant **必须**声明一个 `density` 参数。让该 variant 的 scoped CSS 中所有 spacing tokens 都通过 `calc(var(--p-density, 1) * <base>)` 驱动——包括 paddings、gaps、column widths。这样用户在 airy 和 packed 之间拖动时，布局会重新呼吸，而不需要重新生成。

```json
{"id":"density","kind":"range","min":0.6,"max":1.4,"step":0.05,"default":1,"label":"Density"}
```

如果某个 variant 的拓扑结构确实会变化（如 stacked vs. side-by-side，grid vs. bento），再额外使用一个 `steps` 参数，让 scoped CSS 通过 `:scope[data-p-structure="X"]` 分支。一个 structure 参数 + 一个 density 参数，已经是非常强的组合；不要轻易加第三个。

```json
{"id":"structure","kind":"steps","default":"grid","label":"Structure","options":[
  {"value":"stacked","label":"Stacked"},
  {"value":"grid","label":"Grid"},
  {"value":"bento","label":"Bento"}
]}
```

完整 params contract 见 `reference/live.md`。
