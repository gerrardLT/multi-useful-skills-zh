评估并改进那些显得普通、不一致或结构混乱的排版，让默认味很重的文本变成有判断、有结构的文字系统。

---

## Register

品牌页：按照 [brand.md](brand.md) 里的字体选择流程来。搭配方式应服从品牌赛道，例如 editorial / luxury 用 display serif + sans 正文，tech 往往用一个立场明确的 sans。标题比例用 fluid `clamp()`，层级比例至少 1.25。

产品页：系统字体和常见 sans 栈在这里是合理的。一个调得好的家族通常足以撑起整个 UI。使用固定 `rem` 比例，层级之间更紧，通常在 1.125 到 1.2 之间。

## 评估当前排版

分析当前排版到底哪里弱、哪里普通：

1. **字体选择**
   - 是否在使用“看不见的默认项”？例如 Inter、Roboto、Arial、Open Sans、系统默认字体
   - 字体是否符合品牌人格？一个活泼品牌不该长成企业软件字体
   - 字体家族是否过多？超过 2 到 3 个通常已经开始失控

2. **层级**
   - 你能不能一眼区分标题、正文和说明？
   - 字号是不是挨得太近？例如 14px、15px、16px，只会让层级发糊
   - 字重对比够不够？Medium 对比 Regular 通常太弱

3. **尺寸与比例**
   - 有没有一致的 type scale，还是尺寸全是随手写的？
   - 正文是否达到最基本可读要求？通常至少 16px+
   - 当前尺寸策略是否适合当前场景？应用 UI 用固定 `rem` scale；营销 / 内容页的标题更适合 fluid `clamp()`

4. **可读性**
   - 行长是否舒适？理想范围通常在 45 到 75 个字符
   - 行高是否适合当前字体和场景？
   - 文本和背景之间的对比是否足够？

5. **一致性**
   - 同类元素在整个界面中是否真的长得一样？
   - 字重使用是否一致？同一角色不要这里 bold、那里 semibold
   - letter-spacing 是有意设计过的，还是处处沿用默认值？

**关键：** 目标不是把文字“做得更花”，而是让它更清楚、更好读、更有意图。好的排版往往是隐形的，差的排版才会干扰人。

## 规划排版改进

可以结合 [typography.md](typography.md) 中关于比例、搭配和加载策略的详细说明，一起制定计划。

建立一套系统化方案：

- **字体选择**：需不需要换字体？什么更适合当前品牌和上下文？
- **Type scale**：建立清晰层级的模数比例，例如 1.25
- **字重策略**：哪些角色应该用哪些字重？
- **间距策略**：行高、字距，以及排版元素之间的留白

## 系统性改进排版

### 字体选择

如果需要替换字体：

- 选能体现品牌个性的字体
- 组合出真正有反差的搭配，例如 serif + sans、geometric + humanist；或者只用一个家族，通过不同字重解决
- 确保 webfont 加载不会引起 layout shift，例如 `font-display: swap`，并准备好 metric 匹配的 fallback

### 建立层级

构建清晰的 type scale：

- **5 个字号就能覆盖大部分需要**：caption、secondary、body、subheading、heading
- **层级间使用固定比例**，例如 1.25、1.333 或 1.5
- **组合多个维度**：尺寸 + 字重 + 颜色 + 留白 一起建立层级，而不是只靠尺寸
- **应用 UI**：使用固定 `rem` type scale，必要时只在 1 到 2 个 breakpoint 上微调。过度 fluid 会破坏密集界面需要的空间可预测性
- **营销 / 内容页**：标题和 display text 适合用 `clamp(min, preferred, max)` 做流式缩放，而正文保持固定

### 修正确保可读

- 用 `ch` 单位给文本容器设 `max-width`，例如 `max-width: 65ch`
- 按场景调行高：标题更紧（1.1-1.2），正文更松（1.5-1.7）
- 浅色字放深色背景时，行高可略微再加一点
- 正文至少保持在 16px / 1rem

### 细节打磨

- 数据表和对齐数字使用 `tabular-nums`
- 正确处理 `letter-spacing`：small caps 和大写可略放开，大号 display text 一般默认或略紧
- 用语义 token 命名，例如 `--text-body`、`--text-heading`，不要只写 `--font-16`
- 设置 `font-kerning: normal`，并在合适时考虑 OpenType 特性

### 字重一致性

- 为每种字重定义明确角色，然后坚持使用
- 不要超过 3 到 4 个字重，Regular、Medium、Semibold、Bold 通常已经足够
- 只加载你真的会用到的字重，因为每多一个字重都会增加加载成本

**绝对不要：**

- 使用超过 2 到 3 个字体家族
- 任意挑字号，不建立 scale
- 把正文做到 16px 以下
- 用装饰型 / 展示型字体做正文
- 禁止浏览器缩放（`user-scalable=no`）
- 用 `px` 写字体尺寸，而不是 `rem`
- 在需要个性的地方偷懒回落到 Inter / Roboto / Open Sans
- 搭配两款“差不多但不完全一样”的字体，例如两个几何 sans

## 验证排版改进

- **层级是否清楚**：能不能立刻分出标题、正文、caption？
- **可读性是否提升**：长段正文读起来是否舒服？
- **一致性是否够强**：同角色元素是不是始终长一个样？
- **人格是否成立**：排版有没有体现品牌？
- **性能是否稳妥**：webfont 是否高效加载且不引发布局跳动？
- **无障碍是否达标**：对比度是否满足 WCAG，缩放到 200% 后是否仍然可用？

记住：排版是界面设计的地基，它承载了绝大多数信息。把它做好，是最划算的设计升级。

## Live 模式标志参数

每个 variant 都**必须**声明一个 `scale` 参数，用来控制层级比例。variant 作用域内的所有字号都应该通过 `calc(var(--p-scale, 1) * <base>)` 表达，或者更好地通过 `clamp(min, calc(var(--p-scale, 1) * Npx), max)` 去整体放大 / 收紧字阶。这样用户就可以从更克制一路滑到更有气势。

```json
{"id":"scale","kind":"range","min":0.85,"max":1.3,"step":0.05,"default":1,"label":"Scale"}
```

如果这个 variant 会围绕某组特定字体搭配做变化，那就再暴露一个 `steps` 类型参数，例如 “serif display + sans body”、“mono display + sans body” 或 “all-sans”。不同分支通过 `:scope[data-p-pairing="X"]` 选择器挂到各自作用域 CSS 上。

完整的参数契约见 `reference/live.md`。
