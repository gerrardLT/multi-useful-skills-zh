# 认知负荷评估

认知负荷是用户使用界面时所需的总心理努力。负荷过高时，用户更容易犯错、沮丧并离开。本参考用于识别并修复认知过载问题。

---

## 三种认知负荷

### 内在负荷 — 任务本身
这是用户要完成的事情本身自带的复杂度。你无法消除它，但可以更好地组织它。

**可通过以下方式管理**：
- 将复杂任务拆分为明确步骤
- 提供支架式辅助（模板、默认值、示例）
- 渐进式披露：先展示当前需要的内容，其余先隐藏
- 将相关决策归并在一起

### 外在负荷 — 糟糕设计带来的负荷
这是由糟糕设计决策造成的额外心理负担。**要毫不留情地消除它**，因为它是纯粹的浪费。

**常见来源**：
- 让用户必须自己建立心智地图的混乱导航
- 含义不清、逼用户猜意思的标签
- 互相争夺注意力的视觉杂乱
- 阻碍学习的一致性缺失
- 用户意图与结果之间多余的步骤

### 生成性负荷 — 学习负荷
这是用户为了建立理解而投入的心理努力。这类认知负荷是*有益的*，因为它会带来熟练度和掌握感。

**可通过以下方式支持它**：
- 用渐进式披露逐步暴露复杂性
- 用一致的模式奖励用户学习
- 用反馈确认用户是否理解正确
- 用“做中学”的引导，而不是大段说明文字

## 认知负荷检查清单

用以下 8 项评估界面：

- [ ] **单一焦点**：用户能否在不被其他元素干扰的情况下完成主要任务？
- [ ] **分块**：信息是否以易于消化的组呈现（每组 ≤7 项）？
- [ ] **分组**：相关项目是否通过邻近、边框、共享背景等方式在视觉上归在一起？
- [ ] **视觉层级**：用户能否立刻看出当前屏幕上最重要的内容？
- [ ] **一次只做一件事**：用户能否先聚焦单个决策，再进入下一个？
- [ ] **最少选择**：决策是否已被简化（任一决策点可见选项 ≤5 个）？
- [ ] **工作记忆**：用户是否必须记住前一个页面的信息，才能在当前页面操作？
- [ ] **渐进式披露**：复杂内容是否只在用户需要时才展开？

**评分方式**：统计未通过项。0–1 项未通过 = 认知负荷低（良好）；2–3 项 = 中等（应尽快处理）；4 项及以上 = 高认知负荷（需要优先修复）。

---

## 工作记忆规则

**人类的工作记忆通常一次只能稳定容纳 ≤4 个项目**（Cowan 于 2001 年对 Miller 定律进行了修正）。

在任一决策点，统计用户必须同时考虑的独立选项、动作或信息数量：
- **≤4 项**：仍在工作记忆范围内，可控
- **5–7 项**：已接近边界，考虑分组或渐进式披露
- **8+ 项**：已经过载，用户会跳过、误点或放弃

**实际应用**：
- 导航菜单：顶级项目 ≤7 个（其余归入清晰分类）
- 表单分组：每组视觉断点前可见字段 ≤7 个
- 操作按钮：1 个主操作，1–2 个次操作，其余放进菜单
- 仪表盘卡片：无需滚动即可看到的关键指标 ≤7 个
- 价格档位：选项 ≤3 个（再多会引发选择瘫痪）

## 常见认知负荷违规模式

### 1. 选项墙
**问题**：一次性抛出 10 个以上选择，而且没有层级。
**修复**：按类别分组，突出推荐项，使用渐进式披露。

### 2. 记忆桥
**问题**：用户必须记住第 1 步的信息，才能完成第 3 步。
**修复**：让相关上下文始终可见，或在需要处重复展示。

### 3. 隐藏式导航
**问题**：用户必须自己建立“东西都在哪”的心智地图。
**修复**：始终显示当前位置（面包屑、激活态、进度指示）。

### 4. 术语壁垒
**问题**：技术术语或行业语言迫使用户额外做“翻译”。
**修复**：使用通俗语言；如果必须使用专业术语，就在附近直接解释。

### 5. 视觉噪声底噪
**问题**：所有元素视觉权重都差不多，导致没有重点。
**修复**：建立明确层级：1 个主元素，2–3 个次元素，其余全部弱化。

### 6. 不一致模式
**问题**：相似操作在不同地方的行为不同。
**修复**：统一交互模式。同一类操作 = 同一类界面行为。

### 7. 多任务要求
**问题**：界面要求用户同时处理多个输入（阅读 + 决策 + 导航）。
**修复**：把流程串起来，让用户一次只做一件事。

### 8. 上下文切换
**问题**：用户为了做一个决定，必须在多个页面、标签页或弹窗之间来回切换。
**修复**：把每个决策所需信息尽量放在一起，减少往返跳转。
