# UX Writing

## 按钮文案问题

**不要再用 “OK”、“Submit” 或 “Yes/No” 了。** 它们既偷懒又含糊。优先使用“动词 + 对象”的具体模式：

| 不佳 | 更好 | 为什么 |
|----|----|--------|
| OK | 保存更改 | 明确告诉用户会发生什么 |
| Submit | 创建账户 | 关注结果 |
| Yes | 删除消息 | 明确确认的动作 |
| Cancel | 继续编辑 | 说明“取消”究竟意味着什么 |
| Click here | 下载 PDF | 描述目的地或结果 |

**面对破坏性操作时，一定要把破坏写出来：**

- 用 “Delete”，不要用 “Remove”（前者更偏永久删除，后者更像可恢复）
- 用 “Delete 5 items”，不要只写 “Delete selected”（把数量说出来）

## 错误提示：标准公式

每条错误提示都应回答三个问题：
1. 发生了什么？
2. 为什么会这样？
3. 怎么修？

例如：`邮箱地址无效。请包含 @ 符号。` 就比 `输入无效` 好得多。

### 错误提示模板

| 场景 | 模板 |
|------|------|
| **格式错误** | `“[字段] 需要符合 [格式]。例如：[示例]”` |
| **缺少必填项** | `“请输入 [缺少的内容]”` |
| **权限不足** | `“你无权访问 [对象]。[可替代操作]”` |
| **网络错误** | `“无法连接到 [对象]。请检查网络并尝试 [操作]。”` |
| **服务端错误** | `“我们的服务出现了问题，正在处理。[可替代操作]”` |

### 不要把锅甩给用户

把错误重写成帮助式表达。比如用：
`请输入 MM/DD/YYYY 格式的日期`
而不是：
`你输入了无效日期`

## 空状态是机会，不是空白

空状态其实是引导用户上手的时刻。它至少要做到三件事：

1. 简短承认当前为空
2. 解释为什么把这里填起来是有价值的
3. 给出明确动作

例如：`还没有项目。创建第一个项目开始使用。` 就比单纯一句 `暂无内容` 强得多。

## 语气风格和语调不是一回事

**Voice** 是品牌的人格，应该处处一致。
**Tone** 则要随着情境变化。

| 场景 | 语调变化 |
|------|-----------|
| 成功 | 轻快、简短：`完成了！你的更改已生效。` |
| 出错 | 有同理心、给帮助：`没有成功。你可以试试这些方法……` |
| 加载中 | 安抚式：`正在保存你的内容……` |
| 危险确认 | 严肃、明确：`要删除这个项目吗？此操作无法撤销。` |

**不要拿错误提示开玩笑。** 用户本来就已经烦了，给帮助，不要卖萌。

## 为无障碍而写

**链接文案** 必须脱离上下文也有意义，例如 `查看价格方案`，不要写 `点这里`。**Alt text** 要描述信息，而不是描述“这是一张图”，例如 `第四季度营收增长了 40%`，不要写 `图表`。纯装饰图应使用 `alt=""`。图标按钮必须带 `aria-label`，为读屏器提供语境。

## 为翻译而写

### 预留扩展空间

德语通常比英语长约 30%。界面需要预留空间：

| 语言 | 扩展幅度 |
|------|----------|
| 德语 | +30% |
| 法语 | +20% |
| 芬兰语 | +30-40% |
| 中文 | -30%（字符少，但宽度不一定更小） |

### 更利于翻译的写法

把数字和句子结构拆开考虑。例如 `新消息：3` 通常比 `你有 3 条新消息` 更稳。整句话应尽量作为一个完整字符串，不要把词拆碎，因为不同语言的语序差异很大。避免缩写，例如写 `5 分钟前`，不要写 `5 mins ago`。还要给翻译者上下文，告诉他们这些文案出现在哪里。

## 一致性：术语问题

选一个词，就始终用它：

| 不一致 | 建议统一 |
|--------|------|
| Delete / Remove / Trash | Delete |
| Settings / Preferences / Options | Settings |
| Sign in / Log in / Enter | Sign in |
| Create / Add / New | Create |

建立术语表，并严格执行。术语上的“花样”只会制造困惑。

## 避免冗余文案

如果标题已经说明白了，前言往往就是重复。如果按钮已经足够清楚，就不要再解释一遍。说一次，说准确。

## 加载状态

要具体。用 `正在保存草稿……`，不要只写 `加载中……`。如果等待较长，要么告诉用户预期时间（例如“这通常需要 30 秒”），要么展示进度。

## 确认对话框：少用

多数确认弹窗本质上是设计失败，更好的方式往往是支持撤销。真的必须确认时：

- 把动作写清楚
- 把后果说清楚
- 按钮文案要具体，例如 `删除项目` / `保留项目`，不要只写 `是` / `否`

## 表单说明

格式提示优先放进 placeholder，而不是额外写一段说明。对于那些不显而易见的字段，则要解释为什么要填写。

---

**避免：** 不解释的术语；责怪用户（例如“你出错了”不如“此字段为必填项”）；模糊错误（“出了点问题”）；为了“变化”而乱换术语；在错误里讲笑话。
