# Craft Flow

通过结构化流程构建一个拥有 impeccable 级 UX / UI 质量的功能：先 shape 设计，再确定视觉方向，接着写出真实的生产代码，最后在浏览器里反复检查和改进，直到结果达到高端工作室水准。

## Build Gate

在以下条件全部满足之前，Craft 不能开始构建：

1. PRODUCT 上下文有效且是最新的。
2. 当前任务的 shape 设计 brief 已被用户明确确认，除非用户本来就提供了一份已确认的 brief。
3. brief 中引用到的实现参考资料已加载。
4. shape 的视觉探针决策已记录：已生成、带理由跳过，或已另行解决。
5. north-star mock 的决策已记录：已生成、带理由跳过，或不适用。

PRODUCT.md 和 `teach` 中的回答 **不能** 满足 shape gate。它们只提供项目级上下文。你自己写的一份紧凑 brief 也不能满足 shape gate。`shape=pass` 必须基于一次独立的用户回复，明确批准了 shape brief，或者用户一开始就提供了一份已确认的 brief。

无效的图像跳过理由包括：“最终实现会是语义化 HTML/CSS/SVG”、“图示需要保持可编辑”、“位图 mock 不会被直接用于实现”，或“这是虚构产品”。生成出的 probes 和 mocks 是方向性产物，不是实现资产。

## Craft Contract

Craft 不是一次性初稿。它是一个循环流程，并且必须产出以下工件：

1. 来自 `shape` 的确认版设计 brief。
2. 获批的视觉方向；在图像生成可用时，这个方向应来自已生成的 probes / mocks。
3. Mock fidelity inventory：批准方向中那些必须在代码里保留下来的可见组成要素。
4. 使用项目真实技术栈与约定写出的语义化、可运行实现。
5. 覆盖相关视口的浏览器证据。
6. 首轮浏览器检查之后，至少进行一次 critique-and-fix 循环；除非第一轮就完全没有实质缺陷。

不要让生成的 mockup 取代界面结构、文案、可访问性、响应式行为或状态设计。但必须把已批准的 mock 当作一份具体的视觉契约，用来约束构图、层级、密度、氛围、签名式视觉元素、图像需求和那些真正有辨识度的视觉动作。所谓 “north star” 的意思是“用语义化代码保留关键可见要素”，而不是“拿它当一个模糊 moodboard”。

## 第 1 步：Shape 设计

运行 {{command_prefix}}impeccable shape，并把用户提供的功能描述一并传过去。

在用户完整确认设计 brief 之前，不要进入下一步。brief 就是你的蓝图，所有实现决策都应该能回溯到它。

如果这次 craft 是在 `teach` 创建完 PRODUCT.md 之后恢复的，现在就运行 shape。不要把 teach 访谈、PRODUCT.md，或项目上下文摘要，当成 shape 的替代品。Shape 是任务级的，它必须覆盖范围、内容 / 状态、视觉方向、约束、anti-goals、必要时的 probes，以及明确的 brief 确认。

如果用户已经运行过 {{command_prefix}}impeccable shape，并且手头有一份已确认的设计 brief，就跳过此步，直接使用那份 brief。

## 第 2 步：加载 References

根据设计 brief 中 “Recommended References” 一节，读取相关的 impeccable reference 文件。最低限度，总是要读取：

- [spatial-design.md](spatial-design.md) 处理布局与间距
- [typography.md](typography.md) 处理文字层级

然后根据 brief 的需要追加：
- 交互或表单复杂？读取 [interaction-design.md](interaction-design.md)
- 有动画或过渡？读取 [motion-design.md](motion-design.md)
- 颜色占比重或主题感强？读取 [color-and-contrast.md](color-and-contrast.md)
- 有响应式要求？读取 [responsive-design.md](responsive-design.md)
- 文案、标签或错误状态很多？读取 [ux-writing.md](ux-writing.md)

## 第 3 步：确定视觉方向（按能力决定）

在开始实现之前，当以下条件全部为真时，必须生成高保真的视觉 comps：

- 这项工作是 **全新** 的，或者视觉开放度足够高，先做构图探索会显著提升最终构建质量。
- brief 的目标保真度是 **mid-fi、high-fi，或 production-ready**。
- 当前 harness **原生具备图像生成能力**（例如带原生图片工具的 Codex）。**不要**为了做这一步，要求用户去配置外部 API、shell 脚本或一次性工具链。

只要条件满足，这一步在 Codex 以及任何内建图像生成功能的 harness 中，对 **brand 和 product 工作都同样是强制的**。应使用原生图像生成；在 Codex 中则通过 imagegen skill 使用内建的 `image_gen` 工具。如果图像生成不可用，不要让用户安装 API 或工具。只需用一句话说明由于当前 harness 缺少原生图像生成能力，因此跳过这一步，然后继续。

不要因为“最终 UI 需要语义化、可编辑、代码原生、响应式、可访问”就跳过这一步。这些都是实现要求，不是避免视觉探索的理由。

### 目的

mock 这一步的作用，是找到一个比“直接从代码开始”更强的视觉赛道，而不是靠代码生成去碰运气。brief 在用户、目的、内容、约束、状态和 anti-goals 上仍然是最高权威。mock 只是帮助澄清构图、层级、密度、排版和视觉语气。

### 生成什么

基于已确认的 brief，生成 **1 到 3** 份高保真的 north-star comps。如果 shape 已经产出过方向性 probes，就以它们的结果为输入，在胜出的视觉赛道上做更高分辨率的推进，而不是重新开启另一轮无关探索。

- 对于 brand 工作，要在视觉识别、构图和气质上大胆推进。
- 对于 product 工作，也同样要推进层级、拓扑、密度和语气，但 comps 仍要扎根在真实的产品结构与状态上。
- 对于落地页和长内容 brand 页面，要展示足够多的下一节或第二屏内容，让视觉系统不只停留在 hero。

这些 comps 在主视觉方向上必须真正不同，而不能只是换个配色。

### 批准循环

展示 comps，并询问哪些内容要继续保留下去。如果用户要求修改，或当前方向依然偏弱，那就在进入实现前先生成一轮聚焦修订。一直持续，直到有一个方向被批准，或者用户明确把选择权交给你。

如果用户把选择权交给你，那就选最强的一条，并基于 brief，而不是基于个人口味，解释你的决定。

进入实现前，必须先总结：

- 哪些东西要带进代码
- 哪些东西 **不要** 从 mock 中字面照搬

这个总结是第 4 步之前的必需品。它是视觉探索与语义化实现之间的交接件。

### Mock fidelity inventory

开始构建前，先盘点获批 mock 中的重要可见组成要素：

- Hero 轮廓与主导构图。
- 签名式元素：星球、设备、人像、图表、路径线、嵌套小窗、徽章，或其他令人记住的对象。
- 导航和主 CTA 的处理方式。
- mock 中能看见的章节顺序，尤其是第二屏。
- 这个概念所依赖的图像型内容。
- 排版、密度、颜色 / 材质处理，以及动效线索。

对每个要素，都要决定如何实现：语义化 HTML/CSS/SVG、生成资产、项目现成资源、图标库、canvas/WebGL，或一个被明确接受的省略项。除非用户批准，否则在获批之后，不要擅自替换 hero 构图或另起一种新的视觉驱动方式。

把 mock 当作 **north star**，而不是一张要逐像素描摹的截图。**不要**把核心 UI 文本 rasterize，也不要让 mock 推翻已确认的 brief。但如果最终结果缺失 mock 中那些主要可见要素，那这个实现就是错的。

## 第 4 步：抽取资产（按需要决定）

如果最终选定的方向中包含一些图像原生的视觉要素，而把它们单独导出为资产会显著提升实现质量，就在开始构建前先把它们抽出来。

适合抽取的对象包括：

- stickers
- badges
- seals
- tickets
- graphic labels
- textures
- abstract objects
- decorative marks
- 非语义化场景元素

对于旅游、编辑风、作品集、场馆、产品展示、娱乐、教育，或任何以图像为核心的 brand 页面，视觉资产通常是核心内容，而不是装饰。不要在已批准的 mock 或主题本身明显需要真实图像、生成画面、插画、地图、产品 / 物体渲染或场景图像时，只交付抽象的 CSS 面板。

**不要**为核心 UI 文本、导航、正文文案，或任何本该保持语义化、可编辑的结构导出图像资产。

通常 **1 到 5** 个独立抽取出的资产就够了。如果一个设计可以干净地用 HTML/CSS/SVG 实现，那应优先选择这种方式，而不是用位图资产。反过来，如果 mock 中包含无法用代码可信实现的主要视觉内容，那么资产抽取就不是可选项。

## 第 5 步：构建到生产质量

严格按照设计 brief 实现该功能。分阶段构建，让结构、视觉系统、状态、动效 / 媒体，以及响应式行为都能被有意识地处理。下面这份清单是完成定义，不是灵感提示。

### 生产门槛

- 使用真实或高度可信的内容。向用户展示之前，移除占位文案、占位图、死链接、假控件和无用脚手架。
- 保留获批 mock 中的主要要素。缺失 hero 对象、世界观 / 产品图像、不同的章节结构、被降级的 CTA / 导航处理，或用泛化替代物替换签名式元素，都是阻塞性缺陷，除非用户明确接受了这种改动。
- 先做语义化：真实 heading、landmark、label、表单关联、button / link 语义、可访问名称，以及必要时的状态播报。
- 刻意校准间距、对齐、网格落位和纵向节奏。不要接受默认 gap、随手加的 margin、失衡的留白，或肉眼上的偶然错位。
- 让排版变得有意图：合适的字体加载策略、明确层级、可读 measure、稳定换行、经过调教的换行与包裹方式，以及在移动端和大桌面尺寸下都不出现溢出。
- 补齐真实状态：default、hover（若支持）、focus-visible、active、disabled、loading、error、success、empty、overflow、长文本、短文本，以及必要时的首次使用状态。
- 让交互质量显得完整：键盘路径、触控目标、反馈时机、滚动行为、状态切换过渡，以及避免任何只能通过 hover 才能访问的功能。
- 如项目已有既定图标体系，就用它。如果没有，就选择一套统一的图标库，或使用可访问文本控件；不要混搭不相干的图标风格。
- 优化图像和媒体：正确尺寸、有用的 alt text、首屏以下懒加载、能用时用现代格式、为位图提供响应式 `srcset` / `picture`，并确保项目引用的每个资源都位于工作区内。
- 让动效有高级感：在确实能提升体验时使用 atmosphere blur、filter、mask、shadow 或 reveal 效果；避免随意给布局属性做动画；控制高开销效果；在浏览器中验证流畅度；尊重 reduced motion；避免那种会妨碍任务完成的 choreography。
- 保持可维护性：可复用的本地模式、清晰的组件边界、符合项目约定、不把 UI 文本 rasterize，也不要在更好的本地模式存在时写一次性的硬编码 hack。
- 贴合技术上下文：生产构建通过、没有明显控制台报错、没有可避免的布局抖动、没有无谓依赖、没有坏掉的资源路径。
- 如果你发现某个设计问题会实质性改变 brief 或获批方向，就停下来问，而不是擅自猜。

## 第 6 步：基于浏览器的迭代

**这一步至关重要。** 不要在首轮实现完成后就停下。

把结果在浏览器中打开。在 Codex 里，如果可以，优先用 browser-use 或同类浏览器自动化；否则用 Playwright，或者让用户提供截图。你要检查的是截图，而不只是 DOM 或终端输出。

### 必需的视口检查

检查 brief 所要求的关键视口。默认最低要求：

- 窄屏移动端
- 平板或小尺寸笔记本
- 宽屏桌面端

对每个视口，都要捕获或检查渲染结果，并寻找视觉缺陷：重叠、裁切、层级虚弱、偏离网格的对齐、尴尬留白、拥挤控件、不可读文字、损坏图像、hover-only 功能、布局抖动以及文本溢出。

### Critique and fix 循环

第一轮浏览器检查后，要先给自己写一段简短 critique，然后修补实现。修完之后再次进行浏览器检查。反复循环，直到下面这份清单不再出现实质性问题：

1. **它符合 brief 吗？** 对照设计 brief 的每个部分检查 live 结果。任何不一致都要修。
2. **它符合获批 mock 吗？** 对照 mock fidelity inventory 检查截图：hero 轮廓、主要元素、图像内容、导航 / CTA、章节顺序、密度、颜色 / 材质，以及第二屏内容。缺失关键要素属于 P0 缺陷。
3. **它通过 AI slop 测试了吗？** 如果有人看了说“这就是 AI 做的”，他们会立刻信吗？如果会，那就说明设计意图还不够强。
4. **对照 impeccable 的 DON'T 指南。** 修掉所有反模式违规点。
5. **检查所有状态。** 走一遍 empty、error、loading 和边界情况。每个状态都应显得是有设计过的，而不是事后补丁。
6. **检查响应式行为。** 设计应在构图层面发生适配，而不是只是缩小。
7. **检查工艺细节。** 间距一致性、视觉对齐、文字层级、颜色对比、图像质量、图标一致性、交互反馈、动效时机，以及 focus 处理。
8. **检查性能基础。** 不要出现明显过大的图片、可避免的布局抖动、阻塞型动画，或没有理由的重资产。

退出门槛不是“它能跑”。而是：在所有已检查视口下，渲染结果看起来都经过设计；所有预期状态都已处理；除非用户明确接受，否则没有占位内容残留；并且整体实现质量足以经受高端工作室评审。

## 第 7 步：展示结果

把结果呈现给用户：
- 展示功能的主状态
- 总结你检查过哪些浏览器 / 视口，以及在检查后做了哪些最重要的修正
- 讲解关键状态（empty、error、responsive）
- 解释那些与设计 brief，以及在适用时与所选 north-star mock 相呼应的设计决策。任何已接受的偏离 mock 的地方也要说出来；不要掩盖尚未实现的 mock 要素。
- 诚实说明剩余限制或后续风险
- 最后问一句："What's working? What isn't?"

然后根据反馈继续迭代。好的设计几乎从来不会在第一遍就完全正确。
