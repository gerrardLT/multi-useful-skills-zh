# UX 写作

## 按钮文案问题

**请避免使用“OK”、“Submit”或“Yes/No”。** 这类文案既偷懒又含糊。应优先采用“动词 + 对象”的具体模式：

| 差 | 好 | 原因 |
|----|----|------|
| OK | 保存更改 | 明确告知用户操作结果 |
| Submit | 创建账户 | 聚焦于结果 |
| Yes | 删除消息 | 明确确认的动作 |
| Cancel | 继续编辑 | 说明“取消”的具体含义 |
| Click here | 下载 PDF | 描述目标或结果 |

**对于破坏性操作，务必明确说明其破坏性：**

- 使用“Delete”，而非“Remove”（delete 是永久性的，remove 更偏向可恢复）
- 使用“删除 5 个项目”，而非仅写“删除所选”（需明确数量）

## 错误提示：标准公式

每条错误提示都应回答三个问题：
1. 发生了什么？
2. 为什么会这样？
3. 如何解决？

例如：`邮箱地址无效。请包含 @ 符号。` 远优于 `输入无效`。

### 错误提示模板

| 场景 | 模板 |
|------|------|
| **格式错误** | `"[字段] 需要符合 [格式]。示例：[示例]"` |
| **缺少必填项** | `"请输入 [缺失内容]"` |
| **权限不足** | `"您无权访问 [内容]。[替代操作]"` |
| **网络错误** | `"无法连接至 [服务]。请检查网络连接并 [操作]。"` |
| **服务端错误** | `"系统出现错误。我们正在处理。[替代操作]"` |

### 避免归咎用户

将错误提示改写为帮助性表述。例如使用：
`请输入 MM/DD/YYYY 格式的日期`
而非：
`您输入的日期无效`

## 空状态是机会，而非空白

空状态实则是引导用户入门的时机。它至少应做到三点：

1. 简要说明当前为空
2. 解释填充此处的价值
3. 提供明确操作指引

例如：`尚无项目。创建您的第一个项目以开始使用。` 远优于简单一句 `无项目`。

## Voice 与 Tone 并非同一概念

**Voice（品牌声音）** 是品牌的人格特质，应保持一致。
**Tone（语气）** 则需随情境调整。

| 场景 | 语气变化 |
|------|----------|
| 成功 | 轻快、简洁：`完成！您的更改已生效。` |
| 出错 | 具同理心、提供帮助：`操作未成功。请尝试以下方法...` |
| 加载中 | 安抚式：`正在保存您的工作...` |
| 危险确认 | 严肃、明确：`删除此项目？此操作无法撤销。` |

**切勿在错误提示中开玩笑。** 用户已感困扰，应提供帮助而非卖萌。

## 为无障碍而写

**链接文案** 需脱离上下文仍具意义，例如 `查看定价方案`，而非 `点击此处`。**Alt 文本** 应描述信息内容，而非仅说明“这是一张图”，例如 `第四季度收入增长 40%`，而非 `图表`。纯装饰性图片应使用 `alt=""`。图标按钮必须包含 `aria-label`，为屏幕阅读器提供语境。

## 为翻译而写

### 预留扩展空间

德语通常比英语长约 30%。界面需预留空间：

| 语言 | 扩展幅度 |
|------|----------|
| 德语 | +30% |
| 法语 | +20% |
| 芬兰语 | +30-40% |
| 中文 | -30%（字符数少，但宽度未必更小） |

### 更利于翻译的写法

将数字与句子结构分开考虑。例如 `新消息：3 条` 通常比 `您有 3 条新消息` 更稳定。整句应尽量作为完整字符串，避免拆分词语，因不同语言语序差异较大。避免缩写，例如写 `5 分钟前`，而非 `5 分钟前`。同时需为译者提供上下文，说明文案出现的位置。

## 一致性：术语问题

选定一个词后，应始终使用：

| 不一致 | 一致 |
|--------|------|
| 删除 / 移除 / 丢弃 | 删除 |
| 设置 / 偏好 / 选项 | 设置 |
| 登录 / 进入 / 访问 | 登录 |
| 创建 / 添加 / 新建 | 创建 |

建立术语表并严格执行。术语上的“花样”只会造成混淆。

## 避免冗余文案

若标题已清晰说明，前言往往显得重复。若按钮已足够明确，则无需额外解释。表达一次，表达准确。

## 加载状态

需具体化。使用 `正在保存草稿...`，而非仅写 `加载中...`。若等待时间较长，可告知用户预期时长（例如“通常需要 30 秒”），或展示进度。

## 确认对话框：慎用

多数确认弹窗本质上是设计缺陷，更好的方式往往是支持撤销。确需确认时：

- 明确说明操作内容
- 清晰阐述后果
- 按钮文案需具体，例如 `删除项目` / `保留项目`，而非仅写 `是` / `否`

## 表单说明

格式提示应优先置于 placeholder 中，而非额外添加说明。对于不明显的字段，则需解释询问原因。

---

**避免：** 未解释的术语；归咎用户（例如“您输入有误”不如“此字段为必填项”）；模糊错误（“出现错误”）；为“变化”而随意更换术语；在错误提示中讲笑话。
