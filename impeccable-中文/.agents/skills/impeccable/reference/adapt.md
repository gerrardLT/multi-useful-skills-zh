## 评估适配挑战

先搞清楚：到底要适配什么，为什么要适配。

1. **识别源上下文**：
   - 它原本是为谁设计的？（桌面网页？移动应用？）
   - 当时有哪些默认假设？（大屏？鼠标输入？快速网络？）
   - 当前上下文里，哪些地方做得好？

2. **理解目标上下文**：
   - **设备**：移动端、平板、桌面、电视、手表、打印？
   - **输入方式**：触摸、鼠标、键盘、语音、手柄？
   - **屏幕限制**：尺寸、分辨率、方向？
   - **网络条件**：高速 Wi‑Fi、慢速 3G、离线？
   - **使用情境**：在路上随手看，还是坐在桌前专注阅读？
   - **用户预期**：用户在这个平台上期待什么行为模式？

3. **识别适配难点**：
   - 哪些东西放不下？（内容、导航、功能）
   - 哪些东西不能用？（触屏上的 hover、过小的触控目标）
   - 哪些东西不合适？（把桌面模式硬塞进移动端，或把移动端模式硬塞进桌面）

**CRITICAL**：适配不是单纯缩放，而是为新的上下文重新思考整个体验。

## 规划适配策略

制定真正贴合上下文的策略：

### 移动端适配（Desktop -> Mobile）

**布局策略**：
- 单列替代多列
- 垂直堆叠替代横向并排
- 全宽组件替代固定宽度
- 底部导航替代顶部 / 侧边导航

**交互策略**：
- 触控目标至少 44x44px（不能依赖 hover）
- 在适用场景下支持 swipe 手势（如列表、轮播）
- 用底部抽屉替代 dropdown
- 采用拇指优先设计（控制项位于拇指易达区）
- 增大点击区域，并拉开间距

**内容策略**：
- 渐进披露（不要一次把所有内容全亮出来）
- 优先显示主要内容（次级内容放到 tabs / accordions 里）
- 缩短文案（更精炼）
- 放大字号（至少 16px）

**导航策略**：
- 使用 hamburger menu 或底部导航
- 降低导航复杂度
- 用 sticky header 保持上下文
- 在导航流中加入 back button

### 平板适配（混合策略）

**布局策略**：
- 使用双列布局（不要退化成单列，也别轻易上三列）
- 用侧边面板承载次级内容
- 使用 master-detail 结构（列表 + 详情）
- 根据方向自适应（竖屏 vs 横屏）

**交互策略**：
- 同时支持触摸与指针
- 触控目标仍为 44x44px，但允许比手机更高密度
- 使用侧边抽屉导航
- 在适合时使用多列表单

### 桌面端适配（Mobile -> Desktop）

**布局策略**：
- 使用多列布局（把横向空间利用起来）
- 让侧边导航常驻可见
- 同时展示多个信息面板
- 用 fixed widths + max-width 约束（不要让内容无限拉到 4K）

**交互策略**：
- 用 hover state 展示更多信息
- 增加键盘快捷键
- 在需要时支持右键菜单
- 在有帮助时支持拖拽
- 支持 Shift / Cmd 多选

**内容策略**：
- 一开始就展示更多信息（减少 progressive disclosure）
- 支持更多列的数据表格
- 更丰富的可视化
- 更详细的说明文字

### 打印适配（Screen -> Print）

**布局策略**：
- 在逻辑断点处分页
- 移除导航、页脚和交互元素
- 使用黑白（或有限色彩）
- 预留适合装订的页边距

**内容策略**：
- 展开被截断的内容（完整 URL、原本隐藏的部分）
- 加页码、页眉、页脚
- 补充元信息（打印日期、页面标题）
- 把图表转换成适合打印的版本

### 邮件适配（Web -> Email）

**布局策略**：
- 窄宽度（最大 600px）
- 只用单列
- 使用 inline CSS（不能依赖外部样式表）
- 使用 table-based 布局（为了兼容各种邮件客户端）

**交互策略**：
- 使用大而明确的 CTA（按钮优于文字链接）
- 不依赖 hover（不稳定）
- 对复杂交互使用深链跳回 web app

## 实施适配

系统性地应用改动：

### Responsive Breakpoints

选择合适断点：
- Mobile：320px-767px
- Tablet：768px-1023px
- Desktop：1024px+
- 或者直接使用内容驱动断点（设计在哪里坏，就在哪里断）

### 布局适配技术

- **CSS Grid / Flexbox**：让布局自动重排
- **Container Queries**：根据容器而不是视口进行适配
- **`clamp()`**：让尺寸在最小与最大之间流动变化
- **Media queries**：针对不同上下文使用不同样式
- **Display properties**：按上下文显示 / 隐藏元素

### 触控适配

- 增大触控目标尺寸（至少 44x44px）
- 加大交互元素之间的间距
- 去掉依赖 hover 的交互
- 加入触摸反馈（涟漪、高亮等）
- 考虑 thumb zones（屏幕底部比顶部更容易够到）

### 内容适配

- 谨慎使用 `display: none`（内容虽然隐藏，但仍然会下载）
- 采用 progressive enhancement（先保证核心内容，再在大屏上加增强）
- 对屏外内容做 lazy loading
- 使用响应式图片（`srcset`、`picture`）

### 导航适配

- 在移动端把复杂导航转换成 hamburger / drawer
- 移动应用优先考虑底部导航
- 桌面端保持常驻侧边导航
- 在小屏上加入 breadcrumbs 维持上下文

**IMPORTANT**：一定要在真实设备上测试，而不只是浏览器 DevTools。设备模拟有帮助，但不完美。

**NEVER**：
- 不要在移动端隐藏核心功能（重要功能就必须让它在移动端也能工作）
- 不要默认“桌面端 = 强设备”（还要考虑可访问性和旧机器）
- 不要在不同上下文里改掉整个信息架构（会让用户迷路）
- 不要违背平台预期（移动端用户期待移动端模式）
- 不要忘了移动端 / 平板的横屏
- 不要盲用通用断点（优先用内容驱动断点）
- 不要忽视桌面触控（很多桌面设备也支持触摸）

## 验证适配

在不同上下文中认真测试：

- **真实设备**：真机测试手机、平板、桌面
- **不同方向**：竖屏和横屏
- **不同浏览器**：Safari、Chrome、Firefox、Edge
- **不同系统**：iOS、Android、Windows、macOS
- **不同输入方式**：触摸、鼠标、键盘
- **边界情况**：超小屏（320px）、超大屏（4K）
- **慢网环境**：在限速网络下测试

记住：你是跨平台设计专家。你的目标是让体验在不同上下文下都显得“原生”，同时仍保持品牌和功能的一致性。要有意图地适配，彻底地测试。
