## Register

Brand：色板本身就是声音。先按 SKILL.md 中的策略选定一条颜色路线（Restrained / Committed / Full palette / Drenched），然后严格按该路线控制剂量。Committed、Full palette 和 Drenched 会有意突破“<=10%”规则——那个规则只适用于 Restrained。允许出现意外组合；只要策略需要，一个主导色可以占领整页。

Product：语义优先，且几乎总是 Restrained。强调色只留给主操作、当前选中状态和状态指示器，而不是装饰。每个颜色都必须在所有屏幕里保持稳定语义。

## 评估上色机会

分析当前状态，并找出值得加入颜色的地方：

1. **理解当前状态**：
   - **颜色缺失**：是不是纯灰度？只有中性色？只有一个过于胆小的强调色？
   - **错失机会**：哪里本来可以通过颜色增加意义、层级或愉悦感？
   - **上下文**：针对这个领域和受众，什么样的颜色使用是合适的？
   - **品牌**：有没有现成的品牌色应该延续？

2. **找出颜色能增加价值的地方**：
   - **语义意义**：成功（绿）、错误（红）、警告（黄 / 橙）、信息（蓝）
   - **层级**：吸引注意力到关键元素
   - **分类**：为不同 section、类型或状态赋予区分
   - **情绪语气**：温暖、能量、信任、创造力
   - **Wayfinding**：帮助用户导航、理解结构
   - **Delight**：带来视觉趣味与个性的时刻

如果这些信息从代码库中看不出来，就 {{ask_instruction}}

**CRITICAL**：颜色越多不等于越好。策略性的颜色永远胜过彩虹呕吐。每一个颜色都必须有存在目的。

## 规划颜色策略

建立一套有目的的上色方案：

- **Color palette**：哪些颜色符合品牌 / 场景？（除中性色外，额外颜色控制在 2-4 个）
- **Dominant color**：哪一个颜色应占据 60% 的彩色元素？
- **Accent colors**：哪些颜色用来提供对比和高亮？（30% 和 10%）
- **Application strategy**：每个颜色应该出现在哪，以及为什么出现？

**IMPORTANT**：颜色应该加强层级和意义，而不是制造混乱。关键时刻，少才会更强。

## 有策略地引入颜色

沿这些维度系统性上色：

### Semantic Color
- **状态指示**：
  - Success：绿色系（emerald、forest、mint）
  - Error：红 / 粉系（rose、crimson、coral）
  - Warning：橙 / 琥珀系
  - Info：蓝色系（sky、ocean、indigo）
  - Neutral：灰 / slate 用于 inactive states

- **状态徽章**：用带色背景或边框区分 active、pending、completed 等状态
- **进度指示**：用彩色 bars、rings 或 charts 展示完成度或健康度

### Accent Color Application
- **主操作**：给最重要的按钮 / CTA 上色
- **链接**：让可点击文字有颜色（同时保持可访问性）
- **图标**：给关键图标上色，增加识别度和个性
- **标题 / 小标题**：给 section headers 或关键标签加颜色
- **Hover states**：在交互时引入颜色变化

### Background & Surfaces
- **带色背景**：把纯灰（`#f5f5f5`）换成带温度的中性色（如 `oklch(97% 0.01 60)`），或带冷调的轻微染色（如 `oklch(97% 0.01 250)`）
- **彩色 section**：用微弱背景色区隔区域
- **渐变背景**：用克制但有意图的渐变增加深度，而不是泛化的紫蓝渐变
- **Cards & surfaces**：给卡片或表面增加轻微 tint，带来温度

**使用 OKLCH 进行颜色设计**：它在感知上更均匀，也就是说 lightness 的等距变化看起来也更均匀。非常适合生成和谐色阶。

### Data Visualization
- **Charts & graphs**：用颜色编码分类或数值
- **Heatmaps**：用颜色强度显示密度或重要性
- **Comparison**：通过颜色区分不同数据集或时间段

### Borders & Accents
- **发丝级边框**：使用 1px 的整圈彩色边框，而不是侧边色条（`border-left/right > 1px` 是绝对禁令）
- **下划线**：用有色下划线做强调或 active 指示
- **分隔线**：用轻微带色 divider 替代纯灰线
- **Focus rings**：用品牌相关颜色做 focus 指示
- **Surface tints**：使用 4-8% 的强调色背景洗层，而不是一根色条

**NEVER**：不要使用大于 1px 的 `border-left` 或 `border-right` 作为彩色强调条。这是父级 skill 的三大绝对禁令之一。如果你想把卡片标成 “active” 或 “warning”，就使用整圈发丝边框、背景 tint、前置 glyph，或数字前缀——不要用侧条。

### Typography Color
- **彩色标题**：在保持对比度的前提下，用品牌色给章节标题上色
- **高亮文字**：用颜色表示强调或分类
- **Labels & tags**：用小而明确的彩色标签承载元信息或类别

### Decorative Elements
- **插画**：加入彩色插画或图标
- **形状**：用品牌色几何形作为背景元素
- **渐变**：使用彩色渐变叠层或 mesh backgrounds
- **Blobs / organic shapes**：用柔和的带色有机形状增加视觉兴趣

## 平衡与收束

确保颜色增加的是质量，而不是负担：

### Maintain Hierarchy
- **Dominant color**（60%）：主品牌色或最常用强调色
- **Secondary color**（30%）：用于提供变化的辅助色
- **Accent color**（10%）：用于关键时刻的高对比颜色
- **Neutrals**（剩余部分）：灰 / 黑 / 白承担结构作用

### Accessibility
- **Contrast ratios**：确保符合 WCAG（文本 4.5:1，UI 组件 3:1）
- **不要只靠颜色传达信息**：搭配图标、标签或纹理
- **为色盲做检查**：确保红 / 绿组合不会让部分用户失效

### Cohesion
- **一致的色板**：只使用定义好的 palette，不要随手加色
- **系统化应用**：同一种颜色意义应在全局一致（例如绿色永远 = success）
- **温度一致性**：暖色系保持暖，冷色系保持冷

**NEVER**：
- 不要把彩虹所有颜色都用上（除中性色外，控制在 2-4 种）
- 不要随机上色而不赋予语义
- 不要在彩色背景上放灰字——会显得发灰发脏；应用背景色更深的同色系，或适度透明度处理
- 不要用纯灰做中性色——应加轻微暖 / 冷 tint，才会更有质感
- 不要在大面积区域使用纯黑（`#000`）或纯白（`#fff`）
- 不要违反 WCAG 对比度要求
- 不要让颜色成为唯一信号（这是可访问性问题）
- 不要什么都做成彩色（那会抵消颜色的作用）
- 不要默认回到紫蓝渐变（AI slop 审美）

## 验证上色结果

检查颜色是否真的改善了体验：

- **层级更清楚**：颜色是否在正确引导注意力？
- **意义更清楚**：颜色是否帮助用户理解状态 / 分类？
- **更有吸引力**：界面是否显得更温暖、更愿意让人停留？
- **依然可访问**：所有颜色组合是否都满足 WCAG？
- **不过量**：颜色是否依然平衡且有目的？

记住：颜色既情绪化又有力量。它可以用来制造温度、引导注意力、传达意义和表达个性。但克制和策略，比饱和度和花样更重要。可以大胆，但一定要有意图。

## Live-mode signature params

当从 live mode 调用时，每个 variant **必须**声明一个 `color-amount` 参数，让用户可以在“克制强调”和“颜色淹没表面”之间滑动，而不需要重新生成。variant 的 CSS 应围绕 `var(--p-color-amount, 0.5)` 编写——通常把它作为背景 alpha 的乘数，或作为 OKLCH 表达式中 chroma 轴的缩放因子。0 = 中性 / 单色，1 = 全饱和 / 主导覆盖。

```json
{"id":"color-amount","kind":"range","min":0,"max":1,"step":0.05,"default":0.5,"label":"Color amount"}
```

在此基础上，再叠加 1-2 个 variant-specific 参数：如色板选择（`steps` 命名选项）、色温冷暖、tint 还是 true color。完整 params contract 见 `reference/live.md`。
