# UX 写作

## 按钮文本问题

**不要再使用 “OK”、“Submit” 或 “Yes/No”。** 这些表述既偷懒又含糊。应优先采用“动词 + 对象”的具体模式：

| 差 | 好 | 原因 |
|----|----|------|
| OK | 保存更改 | 明确告知用户将发生什么 |
| Submit | 创建账户 | 聚焦于结果 |
| Yes | 删除消息 | 明确确认的动作 |
| Cancel | 继续编辑 | 说明“取消”具体意味着什么 |
| Click here | 下载 PDF | 描述目的地或结果 |

**对于破坏性操作，务必明确写出破坏性后果：**

- 使用 “Delete”，而非 “Remove”（delete 是永久性的，remove 更接近可恢复）
- 使用 “删除 5 个项目”，而非仅写 “删除所选”（明确说明数量）

## 错误提示：标准公式

每条错误提示都应回答三个问题：
1. 发生了什么？
2. 为什么会这样？
3. 如何修复？

例如：`邮箱地址无效。请包含 @ 符号。` 就比 `输入无效` 好得多。

### 错误提示模板

| 场景 | 模板 |
|------|------|
| **格式错误** | `"[字段] 需要是 [格式]。示例：[示例]"` |
| **缺少必填项** | `"请输入 [缺失内容]"` |
| **权限不足** | `"您无权访问 [内容]。[替代操作]"` |
| **网络错误** | `"无法连接到 [服务]。请检查网络连接并 [操作]。"` |
| **服务端错误** | `"我们这边出了点问题。正在排查中。[替代操作]"` |

### 不要将责任推给用户

将错误提示改写为帮助性表述。例如使用：
`请输入 MM/DD/YYYY 格式的日期`
而不是：
`您输入的日期无效`

## 空状态是机会，而非空白

空状态实际上是用户引导时刻。它至少应做到三件事：

1. 简要承认当前为空
2. 解释为何填充此处有价值
3. 提供明确的操作指引

例如：`尚无项目。创建您的第一个项目以开始使用。` 就比单纯一句 `无项目` 强得多。

## Voice 与 Tone 不同

**Voice** 是品牌的人格，应保持处处一致。
**Tone** 则需随情境变化。

| 场景 | Tone 变化 |
|------|-----------|
| 成功 | 轻快、简短：`完成！您的更改已生效。` |
| 出错 | 具有同理心、提供帮助：`操作未成功。请尝试以下方法...` |
| 加载中 | 安抚式：`正在保存您的工作...` |
| 危险确认 | 严肃、明确：`删除此项目？此操作无法撤销。` |

**不要拿错误提示开玩笑。** 用户本已感到困扰，应提供帮助，而非卖萌。

## 为无障碍而写

**链接文本** 必须脱离上下文也能理解其意，例如 `查看定价方案`，而非 `点击此处`。**替代文本** 应描述信息内容，而非描述“这是一张图”，例如 `第四季度收入增长 40%`，而非 `图表`。纯装饰性图片应使用 `alt=""`。图标按钮必须包含 `aria-label`，为屏幕阅读器提供上下文。

## 为翻译而写

### 预留扩展空间

德语通常比英语长约 30%。界面需预留空间：

| 语言 | 扩展幅度 |
|------|----------|
| 德语 | +30% |
| 法语 | +20% |
| 芬兰语 | +30-40% |
| 中文 | -30%（字符较少，但宽度不一定更小） |

### 更利于翻译的写法

将数字与句子结构分开考虑。例如 `新消息：3` 通常比 `您有 3 条新消息` 更稳定。整句话应尽量作为完整字符串，不要拆分词语，因为不同语言的语序差异很大。避免使用缩写，例如写 `5 分钟前`，而非 `5 分钟前`。还需为译者提供上下文，告知这些文本出现的位置。

## 一致性：术语问题

选定一个词，就始终使用它：

| 不一致 | 一致 |
|--------|------|
| Delete / Remove / Trash | Delete |
| Settings / Preferences / Options | Settings |
| Sign in / Log in / Enter | Sign in |
| Create / Add / New | Create |

建立术语表，并严格执行。术语上的“花样”只会制造困惑。

## 避免冗余文本

如果标题已说明清楚，前言往往就是重复。如果按钮已足够清晰，就无需再解释一遍。说一次，说准确。

## 加载状态

要具体。使用 `正在保存草稿...`，而非仅写 `加载中...`。如果等待时间较长，要么告知用户预期时间（例如 “通常需要 30 秒”），要么展示进度。

## 确认对话框：慎用

多数确认弹窗本质上是设计缺陷，更好的方式往往是支持撤销操作。确需确认时：

- 明确写出操作内容
- 说明操作后果
- 按钮文本要具体，例如 `删除项目` / `保留项目`，而非仅写 `是` / `否`

## 表单说明

格式提示应优先放入占位符，而非额外添加说明文字。对于那些不显而易见的字段，则需解释询问原因。

---

**避免：** 不解释的术语；责怪用户（例如 “您出错了” 不如 “此字段为必填项”）；模糊错误（“出了点问题”）；为了“变化”而随意更换术语；在错误信息中讲笑话。
