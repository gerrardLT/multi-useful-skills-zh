# Color & Contrast

## 色彩空间：使用 OKLCH

**别再用 HSL 了。** 请用 OKLCH（或 LCH）。它是感知均匀的，这意味着明度上相同的步进，视觉上看起来也更接近相同；而在 HSL 中，黄色的 50% 明度会显得很亮，蓝色的 50% 明度却会显得很暗。

OKLCH 函数接收三个分量：`oklch(lightness chroma hue)`。其中 lightness 是 0-100%，chroma 大致在 0-0.4，hue 在 0-360。要构建主色及其更亮 / 更暗的变体，通常应保持 chroma 和 hue 大致稳定，只调整 lightness；但**越接近纯白或纯黑时，越应该降低 chroma**，因为极端明度下的高 chroma 会显得刺眼又俗。

你选择的 hue 应该来自品牌本身，而不是默认反射。不要本能去拿蓝色（hue 250）或暖橙色（hue 60），它们只是 AI 设计中最常见的默认项，不是任何具体品牌的天然答案。

## 构建有功能的色板

### 带倾向的中性色

**纯灰已经过时了。** 完全没有 chroma 的中性色，放在有品牌色的系统里会显得没精神。给所有中性色加入一点非常轻的 chroma（0.005-0.015），并让 hue 轻微朝品牌色方向偏移。这个偏移通常小到不会被用户明确感知为“染色”，但足以在潜意识里让品牌色和界面表面更协调。

你要往哪个 hue 偏，必须来自**这个项目**的品牌本身，而不是“暖色 = 友好，冷色 = 科技”这类公式。如果你的品牌色是 teal，中性色就轻轻往 teal 偏；如果品牌色是 amber，就轻轻往 amber 偏。重点是与**具体品牌**形成一致，而不是套一套通用模板。

**避免**总往暖橙或总往冷蓝偏。这两种是最偷懒的默认项，只会制造另一种单一化。

### 色板结构

一个完整系统通常需要：

| 角色 | 用途 | 示例 |
|------|------|------|
| **Primary** | 品牌、CTA、关键操作 | 1 个颜色，3 到 5 个色阶 |
| **Neutral** | 文本、背景、边框 | 9 到 11 阶的色阶 |
| **Semantic** | success、error、warning、info | 4 种颜色，每种 2 到 3 阶 |
| **Surface** | 卡片、模态、覆盖层 | 2 到 3 个层级 |

**如果不需要，就不要硬加 secondary / tertiary。** 大多数应用一个强调色就够了。额外颜色越多，决策疲劳和视觉噪音越大。

### 正确理解 60-30-10 法则

这个法则说的是**视觉权重**，不是像素占比：

- **60%**：中性色背景、留白、基础表面
- **30%**：次级颜色，例如文本、边框、非激活状态
- **10%**：强调色，例如 CTA、高亮、焦点状态

最常见的错误是：因为某个颜色“是品牌色”，就到处都用。强调色之所以有效，正是因为它稀少。滥用会直接削弱它的力量。

## 对比度与无障碍

### WCAG 要求

| 内容类型 | AA 最低要求 | AAA 目标 |
|----------|-------------|----------|
| 正文 | 4.5:1 | 7:1 |
| 大号文字（18px+ 或 14px 粗体） | 3:1 | 4.5:1 |
| UI 组件、图标 | 3:1 | 4.5:1 |
| 非必要装饰 | 无要求 | 无要求 |

**一个常见坑：** placeholder text 也需要达到 4.5:1。你到处看到的那种浅灰 placeholder，往往都不达标。

### 危险的配色组合

下面这些组合经常会导致对比不足或可读性问题：

- 浅灰字配白底（无障碍里最常见的失败）
- **灰字放在任何彩色背景上**。灰色在色彩背景上会显得发死又没精神。应改用背景色的深色变体，或者用透明度方案
- 红字配绿底，或反过来。相当比例的男性无法清楚区分
- 蓝字配红底，视觉会明显振动
- 黄字配白底，几乎总会失败
- 细字、浅色字直接压在图像上，对比不可预测

### 永远不要用纯灰或纯黑

纯灰（`oklch(50% 0 0)`）和纯黑（`#000`）在自然世界里几乎不存在。真实的阴影和表面总会带一点色相偏移。哪怕 chroma 只有 0.005 到 0.01，也足以让它更自然，而不会明显显得“被染色”。

### 测试

不要只相信自己的眼睛。请使用工具：

- [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/)
- 浏览器 DevTools -> Rendering -> Emulate vision deficiencies
- [Polypane](https://polypane.app/) 实时检测

## 主题：浅色模式与深色模式

### 深色模式不是浅色模式反相

不能简单把颜色倒过来。深色模式需要另一套设计决策：

| 浅色模式 | 深色模式 |
|----------|----------|
| 通过阴影制造层次 | 通过更亮的表面制造层次（少用阴影） |
| 深色文字配浅底 | 浅色文字配深底（适当降低字重） |
| 鲜艳强调色 | 稍微去饱和的强调色 |
| 白色背景 | 不要用纯黑，改用深灰（oklch 12-18%） |

在深色模式下，层次来自表面明度，而不是阴影。建立一个 3 级 surface scale，让更高层级的表面更亮，例如 15% / 20% / 25% lightness。保持与品牌色相同的 hue 和 chroma，只调整明度。正文文字的字重可以稍微降一点（例如 350 而不是 400），因为浅色文字在深底上会视觉上显得更重。

### Token 层级

建议使用两层 token：primitive tokens（例如 `--blue-500`）与 semantic tokens（例如 `--color-primary: var(--blue-500)`）。做 dark mode 时，只需要改 semantic 层，primitive 层保持不动。

## Alpha 往往是一种设计味道问题

大量依赖透明度（rgba、hsla），通常意味着你的色板本身没有定义完整。Alpha 会带来不可预测的对比度、额外性能开销，以及视觉不一致。更好的做法是为不同上下文定义明确的 overlay color。例外情况是焦点环或交互状态，这类场景确实需要半透明。

---

**避免：** 只靠颜色传达信息；搭色时没有清晰角色分工；大面积使用纯黑（`#000`）；跳过色盲测试（约 8% 男性会受影响）。
